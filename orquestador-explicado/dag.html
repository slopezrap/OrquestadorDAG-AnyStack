DAG, Write set y Conflict groups

El DAG (grafo dirigido acíclico) es lo que permite paralelizar slices con seguridad. El Write set y el Conflict group son las dos columnas que impiden que dos terminales se pisen los ficheros. Aquí está la mecánica completa.

01De celda a grafo: el viaje

Todo empieza en una tabla markdown del implementation checklist. bootstrap_source_of_truth.py la lee y la convierte en el grafo. El humano nunca toca el grafo directamente — sólo la tabla.

docs/source-of-truth/<APP>_IMPLEMENTATION_CHECKLIST.md | Slice ID | Depends on | Conflict group | Write set | ... |------------------|-------------------|----------------|-----------------------------| | P01-S01-T001 | — | foundation | api/migrations/** | | P01-S01-T002 | P01-S01-T001 | foundation | api/features/auth/** | | P01-S02-T001 | P01-S01-T002 | front:auth | src/features/auth/** | | P02-S01-T001 | P01-S01-T002 | back:profile | api/features/profile/** | | P02-S01-T002 | P02-S01-T001 | front:profile | src/features/profile/** | │ ▼ bootstrap_source_of_truth.py --refresh │ ▼ orchestrator-state/tasks/registry.json (canónico runtime) orchestrator-state/memory/task-dag.json (grafo derivado) orchestrator-state/memory/task-dag.md (humano) orchestrator-state/memory/execution-graph.json (vista runner)
La fuente canónica del runtime es registry.json → tasks[]. Los task-dag.json y .md son vistas derivadas. check-task-dag.sh --strict verifica que las vistas no han driftado del registry antes de paralelizar.

02Las 4 columnas que controlan el DAG

El Coverage Registry tiene 19 columnas, pero estas 4 son las que el motor consulta para decidir orden y paralelismo.

Depends on

Las dependencias duras. Acepta:

  • — o vacío → nodo raíz
  • P02-S01-T001 → un ID concreto
  • P02-S01-T001..T004 → un rango
  • P03-S02 → todas las tasks de ese step
  • P03 → todas las tasks de esa phase
  • previous → la anterior en la lista

Si falta esta columna entera → bootstrap rechaza arrancar. Sin DAG implícito.

Conflict group

Etiqueta libre que comparten tasks que no deben correr a la vez aunque sus deps estén satisfechas.

Convención común:

  • front:<feature> — mismo módulo frontend
  • back:<feature> — mismo módulo backend
  • migrations — todas las que añaden tablas
  • auth, infra, config — dominios sensibles

Write set

Globs de paths que la task está autorizada a escribir. El hook hook_write_scope_guard.py avisa si una escritura cae fuera; claim_task.py rechaza la task si intersecta con tasks activas.

src/features/login/**
api/features/auth/login.py
api/migrations/0042_*.sql

Journey refs

IDs del journey (de la Journey Coverage Matrix de instrucciones.md).

J104                 # un journey
J104, J201           # dos journeys
—                    # foundation/infra

Cuando todas las tasks de un journey terminan, ese journey queda cerrado y necesita /verify-journey.

04Visualización del grafo

Versión simplificada para ver cómo las dependencias generan waves.

Las waves se calculan por longest path from root: una task pertenece a la wave N donde N = 1 + max(wave de sus deps). Dentro de una wave, las tasks pueden correr en paralelo siempre que sus Conflict groups y Write sets no choquen.

05El comando /next-wave

Script next_wave.py envuelto en ./scripts/next-wave.sh. Read-only sobre el registry.

Qué hace internamente

  1. Encuentra la primera fase incompleta.
  2. Filtra tasks status: ready con todas sus deps done.
  3. Comprueba que no hay pending_journey_verifications.
  4. Comprueba que no hay FU high|critical|blocker proposed.
  5. Busca el subconjunto más grande sin conflict group repetido ni write set intersectado.
  6. Imprime los exports listos para pegar.

Output típico (versión actual, con pr-flow default)

$ ./scripts/next-wave.sh --limit 4

06El claim atómico (claim_task.py)

Antes de empezar una slice, hay que reclamarla. Es la barrera contra dos terminales con el mismo TASK_ID.

claim_task.py P02-S03-T001 │ ▼ file_lock(registry.json) # orden: registry primero │ ├─ ¿task.status ∈ {ready, in_progress}? ├─ ¿deps todas done? ├─ ¿no hay pending_journey_verifications? ├─ ¿no hay FU bloqueantes proposed? ├─ ¿conflict_group no coincide con task in_progress/claimed? ├─ ¿write_set no intersecta con task in_progress/claimed? │ ├─ TODO OK: │ task.status = claimed │ last_claimed_task_id = TASK_ID │ crea task-pack mínimo si no existe │ file_lock(runtime-state.json) # orden: runtime después │ runtime-state.last_worker = caller │ imprime exports │ └─ ALGO FALLA: return error con razón concreta registry intacto
Orden de locks "registry FIRST, runtime-state SECOND" es project-wide. Todos los scripts y hooks lo respetan. Invertir abriría ventana de deadlock.

07Conflict group vs Write set: ¿no es lo mismo?

No. Son dos guardas diferentes que se complementan.

Write set protege ficheros físicos

Glob de paths. Regla: dos tasks no activas si sus globs intersectan. Test mecánico: ¿algún path matchea ambos globs?

Conflict group protege dominios

Etiqueta libre. Regla: dos tasks no activas si comparten conflict_group, aunque sus write sets no se solapen.

Ejemplo: dos slices de auth tocan partes diferentes (login UI vs register UI) pero ambas importan src/shared/auth/token_store.ts. conflict_group: auth las serializa.

El bug que un Conflict group previene

# Sin conflict group:
P02-S01-T001 (login UI)     write_set: src/features/auth/login/**
P02-S01-T002 (register UI)  write_set: src/features/auth/register/**

# Globs NO intersectan. /next-wave los propone en paralelo.
# Ambos importan src/shared/auth/token_store.ts.
# Si login cambia firma y register no lo sabe, frontend se rompe.

# Con conflict group:
P02-S01-T001  conflict_group: auth   write_set: src/features/auth/login/**
P02-S01-T002  conflict_group: auth   write_set: src/features/auth/register/**

# /next-wave se niega a proponerlos en paralelo. Una primero, otra después.

08Cumulative product: v0 + v1 + v2 + ...

El Coverage Registry no se reescribe. Crece. Product increment y Build state marcan qué filas ya están construidas.

| Slice ID    | Product increment | Build state | ...
| P01-S01-T001 | v0 (baseline)    | done        |   # ya construida
| P03-S01-T001 | v1               | done        |   # incremento previo
| P04-S01-T001 | v2               | planned     |   # incremento actual
| P04-S01-T002 | v2               | planned     |

El closer, tras cerrar la última slice de un incremento, ejecuta ./scripts/sync-product-baseline.sh sync --version v2 --task P04-S99-T999. Copia docs/source-of-truth/ a docs/product-baseline/ con manifest SHA-256.

09Journeys: el otro tipo de dependencia

Además del DAG de tasks hay un grafo paralelo: los journeys. Recorridos del usuario que cruzan varias slices.

Cuándo se cierra un journey

Cuando todas las tasks que lo referencian están done. journeys_closing_at_task en common.py lo calcula status-based.

Qué hace el cierre

  • Inline (§5.bis "ahora") → JOURNEY_VERIFIED_INLINE: J104 → hook marca journey.verification_status = verified.
  • Aparte → JOURNEY_PENDING_VERIFY: J104 → hook lo añade a runtime-state.pending_journey_verifications.

Cómo bloquea el DAG

Con un journey pendiente, /next-wave y claim_task.py filtran las tasks que lo referencian. El resto del DAG sigue.

10Granularidad sin topes artificiales

La política actual elimina los límites numéricos de phase/step/slice. El DAG se valida por estructura, dependencias, ownership y trazabilidad.

UnidadQué esTamaño sano
PhaseMilestone o módulo completoTodas las slices necesarias
StepLane coherente: pantalla/journey, módulo dominio, foundation laneSin hard cap; agrupa por coherencia y verificación
Slice/TaskUnidad ejecutable y verificable por un worker en un terminal1 slice = 1 acceptance verificable
Anti-patrón: dividir por capas (backend → frontend → UX polish). Cada pantalla importante debe cubrir contrato + API/datos + UI conectada + estados UX + verify de journey, todo en una slice.
