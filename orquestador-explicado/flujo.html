El flujo per-slice, paso a paso

Una "slice" es una fila del Coverage Registry: una tarea atómica con dependencias, write set y verificación. Toda slice atraviesa la misma cadena de subagentes, gates y commits. Aquí está el viaje completo, desde abrir terminal hasta cerrar la tarea.

01Qué pasa antes de invocar al primer agente

El pipeline no arranca sin contexto. Antes de /next-slice hay tres pre-requisitos.

① El proyecto bootstrapeado

Los 5 docs source-of-truth existen y son coherentes. bootstrap_source_of_truth.py --refresh ya generó registry.json y task-dag.json.

② Servidores arriba

Backend (curl /health → 200), frontend dev server o emulador según TECHNICAL_GUIDE, y base de datos con migraciones aplicadas. Si algo está caído, no se trabaja.

③ Terminal scoped + worktree del TASK_ID

Has elegido un TASK_ID con /next-wave y, antes de arrancar Claude, te has movido al checkout correcto (en pr-flow el worktree creado por ensure-task-worktree.sh; en push-to-main la rama main):

cd "$(./scripts/ensure-task-worktree.sh P02-S03-T001)"
export CLAUDE_ACTIVE_TASK_ID=P02-S03-T001
export CLAUDE_TASK_PACK=orchestrator-state/tasks/
  task-packs/P02-S03-T001.md
Cambio reciente · worktree session-level: antes el aislamiento Git lo declaraba cada subagente con isolation: worktree. Ahora es el terminal worker entero el que entra en el worktree del TASK_ID, vía ./scripts/ensure-task-worktree.sh. Todos los subagentes ven el mismo checkout — validator/tester pueden inspeccionar lo que escribió developer, closer commitea desde ahí. Más detalle en la página de agentes.

02El comando que dispara todo: /next-slice

Es un slash command definido en .claude/commands/next-slice.md. Recibe el TASK_ID y orquesta los pasos 1 a 4 del pipeline. Después de tester pass continúa automáticamente con verify-slice y no llama al closer.

$ claude --agent main-orchestrator --permission-mode bypassPermissions \ "/next-slice P02-S03-T001" ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ # Sesión arrancada como main-orchestrator # SessionStart hook inyecta: PROGRESS.md head, journeys pendientes, blockers # El comando lee task-pack y orquesta los agentes

El flag --permission-mode bypassPermissions evita prompts de aprobación. Es seguro aquí porque los hooks del proyecto enforcen scope, presupuesto y trailers; el bypass no anula los hooks.

03El pipeline completo, paso a paso

Cada bloque es una etapa. Las etapas con borde violeta corren dos agentes en paralelo (un solo mensaje con dos llamadas Agent). El borde naranja marca un gate humano. El borde verde marca una etapa de cierre con lifecycle.

1planner

opus[1m] · maxTurns 80 · blocking · sin worktree

Es el primer agente y es blocking: si emite CONTEXT_READY: no, todo el pipeline se detiene aquí.

Qué hace

  • Lee registry.json y verifica que el TASK_ID está ready y sus deps están done.
  • Comprueba que no hay journeys pendientes ni FU bloqueantes que afecten a la slice.
  • Extrae de los 5 source-of-truth docs sólo las secciones relevantes (acceptance, verify, write set, allowed_paths, endpoints, tablas, journey refs).
  • Las consolida en orchestrator-state/tasks/task-packs/<TASK_ID>.md.
  • Decide si necesita docs oficiales (NEEDS_OFFICIAL_DOCS: yes) según el stack y la slice.

Salida

OUTCOME: ready
CONTEXT_READY: yes
NEEDS_OFFICIAL_DOCS: yes
IMPACT_READY: yes
JOURNEY_SCOPE: J104
CLOSES_JOURNEY: none

2developer (+ official-docs-researcher si aplica)

developer obligatorio · official-docs-researcher sólo si planner marca NEEDS_OFFICIAL_DOCS

El main-orchestrator invoca siempre developer. Sólo añade official-docs-researcher si el planner marcó NEEDS_OFFICIAL_DOCS: yes o la slice toca APIs/librerías externas no confirmadas; en ese caso pueden ir en el mismo mensaje con dos Agent tool calls.

developer (sonnet[1m] · maxTurns 150 · worktree)

Trabaja en un git worktree aislado. Implementa en orden estricto: DB → backend → frontend. Cada función con logging BEFORE/AFTER. Actualiza PROGRESS.md e inicializa el handoff con sus secciones.

official-docs-researcher (sonnet[1m] · maxTurns 50 · background)

Sólo si NEEDS_OFFICIAL_DOCS: yes. Recibe 1-5 preguntas concretas, usa cache + MCPs antes de WebFetch oficial. Si encuentra discrepancia con los docs internos, escribe nota en memory/official-doc-notes/<tema>.md con campo RESOLVED: pendiente.

El siguiente Write/Edit del developer dispara hook_docs_discrepancy_check.py, que avisa de la discrepancia sin resolver. El developer debe reconciliar (actualizar task pack y poner RESOLVED: <fecha>) antes de continuar.

Trailer del developer

TASK_ID: P02-S03-T001
OUTCOME: success
NEXT_STATUS: validator_tester_pending
HANDOFF: orchestrator-state/tasks/handoffs/P02-S03-T001.md
EVIDENCE: orchestrator-state/tasks/evidence/P02-S03-T001/

3validator ‖ tester

paralelo · gana tester en lifecycle, validator info-only

Misma mecánica que el paso 2: un mensaje, dos Agent calls. Pero aquí el conflicto al cerrar es real.

validator (opus[1m] · maxTurns 60 · info-only)

Lee el diff y el handoff. Comprueba arquitectura, file size, logging, security, journey matrix, wiring. No corre tests. No toca código. Apendea sección ## validator al handoff con su OUTCOME.

tester (sonnet[1m] · maxTurns 80 · lifecycle)

Corre tests reales contra backend+DB+frontend levantados. Smoke curl. Verifica logs con ambos modos de verbosidad. Guarda outputs en evidence/<TASK_ID>/. Apendea sección ## tester al handoff.

Quién manda en el status

Cuando ambos terminan, el hook SubagentStop adquiere file_lock(registry.json) y, según el orden de llegada:

  • Si llega tester primero → escribe task.status = ready_for_close o needs_debug.
  • Si llega validator primero → escribe task.validator_outcome y validator_next_status como metadatos (info-only).
  • El segundo en llegar respeta la regla de su rol y el resultado final es deterministico: tester gana lifecycle, validator gana metadata.

Qué decide el siguiente paso

Camino feliz

tester pass + validator approved → pipeline pausa. Tester pass es el punto de salida de /next-slice.

Camino con defecto

tester fail o validator changes_requested → invoca debugger (paso 4) y vuelve a este paso 3.

4debugger (condicional)

opus[1m] · maxTurns 120 · worktree · sólo si hay defecto in-scope

Aplica el fix mínimo dentro del mismo write_set de la slice. Si el defecto exige cambiar scope (nueva ruta, nueva tabla, nueva pantalla), no fixea — registra follow-up formal.

Las 3 reglas duras

  • Máximo 3 ciclos de debugger por TASK_ID. El hook lo cuenta en el handoff.
  • 4º intento → OUTCOME: blocked · reason: max_debug_cycles_reached. Decisión humana.
  • Si el fix sale del write_set declarado → no fix, se crea FU con register-followup-task.sh propose.

Tras el fix, el pipeline vuelve al paso 3: validator ‖ tester de nuevo. Si pasan, sale a tester pass.

/next-slice continúa automáticamente hacia /verify-slice. Si validator/tester pasan, el gate humano se ejecuta en el mismo flujo y reconstruye contexto desde disco. Si la verificación queda verified_pending_close, el siguiente paso deliberado es que el usuario invoque /closer TASK_ID.

5/verify-slice — el gate humano

comando slash · resiliente a /clear · hard reset + datos reales

El comando /verify-slice TASK_ID hace algo radical: reconstruye todo el contexto desde disco. Lee PROGRESS.md, registry.json, runtime-state.json, el handoff, el task pack y el TECHNICAL_GUIDE. No depende del contexto previo de Claude.

Qué pide

  • Hard reset del entorno (dev-restart.sh + DB clean).
  • Carga del Verification Data Contract: datos reales o proporcionados, nunca mocks.
  • Reproducción humana en navegador (o emulador) del flujo front→back→DB.
  • Vigilancia de logs browser/front/back/DB/worker/Docker/Rancher: backend, frontend, DB.
  • Validación de la tabla del task pack: ¿cada celda acceptance se cumple?

Subagente lifecycle: slice-verifier

/verify-slice no verifica por narrativa del main thread: primero escribe un skeleton con init-verify-slice-handoff.sh y después lanza slice-verifier. Este subagente debe usar Chrome DevTools MCP, claude-in-chrome o Agent360 Browser MCP (`browser-mcp`); si ninguno está conectado, bloquea con browser_mcp_unavailable y no cierra por API-only.

Reviewer especial: screen-journey-reviewer

Si la slice toca pantalla, UX, journey o tiene VISUAL_CONTRACT_CHECK, /verify-slice invoca screen-journey-reviewer (info-only) después del slice-verifier y antes del /closer manual. Apendea ## Screen/Journey review al handoff con los estados UI obligatorios y notas a11y. Si el review pide cambios, vuelve a debugger; no es FU.

§5.bis · Journey-closing inline

Si la slice cierra al menos un journey de la matriz, el comando te pregunta:

"Ahora"

Aprovecha el entorno cargado y verifica el journey end-to-end inline. Apendea ## verify-journey al mismo handoff. El closer leerá JOURNEY_VERIFY_OUTCOME: verified y emitirá JOURNEY_VERIFIED_INLINE. Un solo gate.

"Aparte"

Deja el journey pendiente. El closer emite JOURNEY_PENDING_VERIFY: <JID> y el planner difiere futuras slices que referencien ese journey hasta que ejecutes /verify-journey <JID>.

Qué emite verify-slice

## verify-slice
TASK_ID: P02-S03-T001
MCP_BROWSER: chrome-devtools     # o claude-in-chrome/agent360-browser-mcp/browser-mcp
VERIFY_OUTCOME: verified         # issues_found | blocked
DATA_CONTRACT_ROWS: VDC-001,VDC-004
DATA_SETUP: cuenta_sandbox_42, dataset_v2
PERSISTED_DATA_OBSERVED: users/42 active
FLOWS_TESTED: login happy path, error_network
EVIDENCE: orchestrator-state/tasks/evidence/P02-S03-T001/verify-*
verified

mueve a verified_pending_close y, si el checker pasa, deja la slice lista para que ejecutes /closer <TASK_ID> (paso 6).

issues_found

spawnea debugger → vuelve al paso 3.

6closer — commit atómico

sonnet[1m] · maxTurns 50 · checkout de la sesión · lifecycle owner

Lo invocas tú con /closer <TASK_ID> cuando el router de verify deja la task en verified_pending_close: handoff verified completo y checker mecánico verde. Hace 7 cosas en orden:

  1. Pre-check del handoff con check-handoff-contract.sh. Exige las secciones machine-readable: validator approved, tester pass, verify-slice verified. Para slices UI/journey, también Screen/Journey review approved.
  2. Pre-check Git: ejecuta ./scripts/ensure-task-worktree.sh --check-current <TASK_ID>. En pr-flow exige estar en el worktree/rama dev/<TASK_ID>; en push-to-main exige rama main. Si está en otro checkout, bloquea: no cambia de rama por su cuenta.
  3. Escribe el report en orchestrator-state/tasks/reports/<TASK_ID>.md consolidando handoff + evidence.
  4. Sincroniza el baseline con ./scripts/sync-product-baseline.sh sync --version <vN> --task <TASK_ID>. Esto actualiza docs/product-baseline/ como snapshot acumulativo para futuros increments.
  5. Commit atómico en el checkout actual (main en push-to-main, dev/<TASK_ID> en pr-flow). Sin Co-authored-by: Claude, sin git stash. Si hay trazas tardías (ledger, runtime-state) se integran con git commit --amend --no-edit antes del push.
  6. Push + auto-merge con ./scripts/git-workflow.sh, que delega al plugin declarado en STACK_PROFILE. En pr-flow, pr-flow.sh hace push, crea/reusa PR y activa auto-merge squash con --auto; sólo usa admin merge si defines explícitamente CLAUDE_PR_FLOW_ADMIN_MERGE=1. En push-to-main, push directo a main como antes.
  7. Housekeeping: cleanup-slice-runtime.sh --task <TASK_ID> --apply --strict + slice-clean.sh + cleanup-worktrees.sh --apply --task <TASK_ID>.

Trailer del closer (todas las pruebas obligatorias)

TASK_ID: P02-S03-T001
OUTCOME: committed
NEXT_STATUS: done
REPORT: orchestrator-state/tasks/reports/P02-S03-T001.md
REPORT_READY: yes
BASELINE_SYNC_READY: yes
GIT_READY: yes
PUSH_READY: yes
RUNTIME_CLEANED: yes
WORKTREES_CLEANED: yes
JOURNEY_VERIFIED_INLINE: J104    # si aplica
El guardrail final: aunque emita committed/done, si falta cualquiera de las 5 pruebas *_READY=yes, el hook SubagentStop sobrescribe el resultado a blocked. No hay done sin las 5 marcas.

04El handoff: el fichero que cuenta la historia

Todo lo que ocurre en una slice queda registrado en un único fichero: orchestrator-state/tasks/handoffs/<TASK_ID>.md. Crece a lo largo del pipeline. Nadie sobrescribe — sólo apendea su sección.

orchestrator-state/tasks/handoffs/P02-S03-T001.md # Handoff P02-S03-T001 ## developer ← lo escribe developer al terminar OUTCOME: success NEXT_STATUS: validator_tester_pending Files: src/features/x/... Migrations: 0042_add_x.sql Logs verified: yes (both modes) ## validator ← lo escribe validator (info-only) OUTCOME: approved Findings: none Architecture: clean File size: ok (largest 187 lines) ## tester ← lo escribe tester (lifecycle owner) OUTCOME: pass NEXT_STATUS: ready_for_close Backend tests: 14 passed Frontend tests: 9 passed Curl: 200 OK Evidence: orchestrator-state/tasks/evidence/P02-S03-T001/ ## verify-slice ← lo apendea /verify-slice tras gate humano VERIFY_OUTCOME: verified DATA_USED: cuenta_sandbox_42 HUMAN: yes Screenshots: 6 files ## Screen/Journey review ← sólo si la slice es UI/journey OUTCOME: approved UI states: loading/empty/error_network/success all present A11y: keyboard nav ok, alt text ok ## verify-journey ← sólo si rama "ahora" en §5.bis JOURNEY_ID: J104 JOURNEY_VERIFY_OUTCOME: verified SLICES_COVERED: P02-S03-T001, P02-S03-T002, P02-S03-T003 MARGINAL_STATES_TESTED: back, reload, deep_link, error_network

El closer lee este fichero entero para decidir si puede commitear. check-handoff-contract.sh valida que están las secciones requeridas con los enums esperados antes de invocar /closer.

05Resiliente a /clear: el truco operacional

Una slice consume ~150-200k tokens entre planner, developer, validator, tester y posibles ciclos de debugger. Si entras al gate humano con todo eso en contexto, gastas batería y latencia para nada: /verify-slice no necesita ese contexto.

La secuencia recomendada

# Terminal worker, ya con CLAUDE_ACTIVE_TASK_ID exportado claude> /next-slice P02-S03-T001 ... planner → developer → validator‖tester → tester pass ✓ # Liberas ~150k tokens claude> /clear # /verify-slice reconstruye TODO desde disco claude> /verify-slice P02-S03-T001 ... lee PROGRESS.md + registry + handoff + task pack + TECHNICAL_GUIDE ... pide hard reset + datos reales + reproducción → si verified: queda lista para /closer → si issues: spawnea debugger

El comando es idempotente en cuanto a reconstrucción de contexto: aunque corras Claude desde otra máquina con otra sesión, mientras tengas los mismos ficheros en disco y el mismo TASK_ID exportado, el verify-slice ve exactamente lo mismo.

06Multi-terminal: cómo se coordinan varias slices a la vez

Si la wave actual del DAG tiene 3 tasks paralelizables sin conflicto, abres 3 terminales. Cada uno con su propio CLAUDE_ACTIVE_TASK_ID. No hay notificaciones push entre ellos — la sincronización es por disco.

Terminal A Terminal B Terminal C TASK_ID=P02-S03-T001 TASK_ID=P02-S03-T002 TASK_ID=P02-S03-T003 /next-slice P02-S03-T001 /next-slice P02-S03-T002 /next-slice P02-S03-T003 │ │ │ ▼ ▼ ▼ tester pass tester pass tester pass /verify-slice /verify-slice /verify-slice closer commit closer commit closer commit │ │ │ └─ hook actualiza └─ hook actualiza └─ hook actualiza registry.status=done registry.status=done registry.status=done bajo file_lock bajo file_lock bajo file_lock En cualquier momento, otro terminal puede ejecutar: ./scripts/next-wave.sh --limit 4 para ver qué tasks se han desbloqueado tras los cierres.

El claim_task.py protege los terminales entre sí: antes de empezar, comprueba bajo lock que (a) las deps están done, (b) no hay journeys pendientes, (c) no hay FU bloqueantes, (d) el Write set de esta task no choca con tasks in_progress ni claimed por otro terminal. Si choca, devuelve error y no corrompe el DAG.

07Variantes del flujo

El pipeline básico es uno solo, pero hay 4 caminos alternativos según el caso.

/auto-verify-slice — saltarse el gate humano (sólo low+auto)

Para slices con Risk level: low + Verify mode: auto que no cierran journey. Ejecuta las verification_commands del task pack automáticamente, genera evidencia JSON y apendea ## verify-slice. Útil para migraciones, foundation, code-only sin UI.

Rechaza: UI, journey-closing, auth, endpoints mutantes, pagos, datos personales, severity medium+.

/verify-journey — gate de journey en modo "aparte"

Cuando una slice cierra journey y elegiste "aparte" en §5.bis. El closer emite JOURNEY_PENDING_VERIFY: <JID>, el hook lo añade a runtime-state.pending_journey_verifications, y el planner difiere futuras slices que referencien ese journey.

Cuando ejecutas /verify-journey <JID>: hard reset + datos consolidados de TODAS las slices del journey + reproducción end-to-end + estados marginales obligatorios (back, reload, empty, error_network, permission_denied, deep_link, next_action). Escribe journey-handoffs/<JID>.md y trailer JOURNEY_VERIFY_OUTCOME que el hook procesa para limpiar el pending.

/revise-slice — corrección sobre slice canónica

Cuando una slice ya cerrada necesita ajustes (típicamente tras una verify-journey aparte que descubre un problema). NO crea un TASK_ID nuevo, NO cambia el DAG. Apendea ## revision debugger section-section al mismo handoff, evidence en evidence/<TASK_ID>/revision-*, report en reports/<TASK_ID>-revision-<ts>.md. El closer correctivo commitea sobre el mismo TASK_ID.

/phase-gate — cierre real de fase

Antes de abrir la siguiente fase, ejecuta ./scripts/phase-gate.sh <PHASE_ID>. Bloquea si: faltan tasks done, hay handoffs/evidence/reports ausentes, hay journeys pendientes/no waived, hay FU high|critical|blocker abiertas. Con --require-git-clean exige árbol limpio.

08Fallos típicos y qué hacer

Estas son las situaciones reales que te vas a encontrar y cómo el orquestador las maneja.

tester falla 3 veces seguidas

El debugger emite OUTCOME: blocked · reason: max_debug_cycles_reached. El pipeline se detiene. Tienes que mirar el handoff y la evidencia para entender por qué; probablemente el scope estaba mal y necesitas /revise-slice o una FU.

verify-slice encuentra issue

Spawnea debugger automáticamente con scope verify. Tras el fix vuelve a validator ‖ tester y a /verify-slice. Si el issue está fuera del write set → no fix, FU formal.

el push falla en cleanly

El closer ejecuta ./scripts/git-workflow.sh. Si el plugin destino falla, el closer emite PUSH_READY: no → el hook marca blocked. No hay fallback a git push manual: el cierre se trata como fallo y necesita reparación.

spawn budget agotado (21 spawns)

El hook hook_spawn_budget.py devuelve permissionDecision: deny en el spawn 21. La slice debe cerrarse o bloquearse con lo que tiene. Indica que el pipeline está ineficiente — revisa por qué se gastaron 20.

doc oficial contradice tu source-of-truth

El researcher escribe una nota en memory/official-doc-notes/. El hook docs-discrepancy alerta en el siguiente Write del developer. Reconcilia los docs internos y pon RESOLVED: en la nota antes de seguir.

otro terminal tomó la task

claim_task.py bajo lock devuelve error. Significa que esa task ya está reclamada en otro worktree. Elige otra con /next-wave.
