Hooks y locks: las reglas mecánicas

Si los agentes son quién hace el trabajo, los hooks son qué pueden y no pueden hacer. Son scripts que Claude Code dispara automáticamente. Los locks POSIX (fcntl.flock) garantizan que cuando varios agentes terminan a la vez, el estado del proyecto queda consistente.

01Los 4 tipos de hooks de Claude Code

TipoCuándo dispara¿Bloquea?Salidas posibles
PreToolUseAntes de cualquier herramientaSÍpermissionDecision: deny, additionalContext
PostToolUseDespués de la herramientaNoadditionalContext, registros
SubagentStopSubagente terminaNoMutar estado, logs, additionalContext
SessionStartInicio sesión Claude CodeNoadditionalContext
Sólo PreToolUse puede bloquear. Los demás son informativos o de auditoría.

02Cómo se declaran en settings.json

{
  "agent": "main-orchestrator",
  "permissions": { "defaultMode": "bypassPermissions" },
  "hooks": [
    { "event": "PreToolUse", "matcher": {"toolNames": ["Agent"]},

}

03Los 5 hooks reales del orquestador

① hook_spawn_budget.py

PreToolUse · matcher: Agent · BLOQUEA

Cuenta subagentes completados para CLAUDE_ACTIVE_TASK_ID y rechaza el spawn 21 (budget 20 por slice).

Qué lee
  • runtime-state.json (counters por TASK_ID)
Devolución en spawn 21
{
  "permissionDecision": "deny",
  "reason": "Spawn budget agotado
            (20/20)"
}

Por qué: sin este límite, un agente podría entrar en bucle (spawn → fail → debugger → spawn → ...) y consumir todo el budget.

② hook_write_scope_guard.py

PreToolUse · Write/Edit/MultiEdit/NotebookEdit · BLOQUEA

El guardián principal. Bloquea 6 familias de escrituras peligrosas.

Qué bloquea
  • Edits en .claude/ con TASK_ID activo
  • Cross-task: handoff/evidence/report de otro TASK_ID
  • Edits en docs/source-of-truth/ con TASK_ID activo
  • Edits directos en docs/product-baseline/
  • Edits directos a core state: registry.json, runtime-state.json, task-dag.*, ledgers
  • [Nuevo] Edits a scripts/dev-restart.profile.sh y scripts/dev-restart.sh durante slice. Evita que un closer paralelo aplaste el profile concreto con los stubs neutrales del meta-orquestador.
Excepciones controladas
  • CLAUDE_ALLOW_BASELINE_SYNC_WRITES=1 → permite baseline
  • CLAUDE_ALLOW_STATIC_CONFIG_WRITES=1 → para mantenimiento
  • CLAUDE_ALLOW_DEV_PROFILE_WRITES=1 → permite editar dev-restart.profile fuera de slice
Qué avisa (no bloquea)
  • Write fuera del write_set declarado: anota warning, sigue

③ hook_docs_discrepancy_check.py

PreToolUse · Write/Edit · WARN-ONLY

Si el researcher escribió notas en memory/official-doc-notes/ sin RESOLVED:, este hook avisa antes del siguiente write del developer. Nunca bloquea.

Excluye writes a la propia carpeta de notas, a source-of-truth y a handoffs (para no aullar mientras reconcilias).

④ hook_update_ledger.py

PostToolUse · auditoría

Registra todos los eventos de modificación. Dos ledgers separados:

Ledger versionado

ledger.jsonl → eventos Write/Edit/MultiEdit/NotebookEdit. Versionado por Git.

Ledger no versionado

bash-ledger.jsonl → eventos Bash. Ignorado por Git.

¿Por qué separados? El closer hace git push (un Bash) → dispara PostToolUse → escribe en ledger. Si fuera versionado, ensuciaría el árbol justo después del push, rompiendo el invariante "tras push, working tree limpio".

⑤ hook_capture_subagent_stop.py

SubagentStop · MUTA BAJO LOCK

El corazón del estado del proyecto. 7 fases:

  1. Parse trailer del último mensaje del agente (línea-prefijo o JSON fenced).
  2. [Nuevo] Fallback al handoff si stdin trailer vacío: si el mensaje del agente se truncó (token cap, network drop, model stop), el hook lee el bloque CLAUDE_TRAILER del handoff en disco como recuperación. Valida con AGENT: y TASK_ID: que no es un trailer viejo del fichero acumulativo. Logea recovered_from_handoff para que SessionStart lo surfacee.
  3. Valida contra schema: lee trailer_schema.roles.<agente>. Si OUTCOME no está en outcome_values, registra error y NO muta.
  4. Adquiere locks: file_lock(registry) primero, file_lock(runtime-state) después.
  5. Decide info-only vs lifecycle: validator → metadata; tester → task.status.
  6. Guardrails: si closer dice done sin las 5 pruebas, reescribe a blocked.
  7. Procesa journey trailer: JOURNEY_PENDING_VERIFY, JOURNEY_VERIFIED_INLINE.

Mismatch de TASK_ID: si el trailer reporta TASK_ID distinto de CLAUDE_ACTIVE_TASK_ID, niega la mutación pero conserva la nota en ledger.

Sobre el fallback handoff: el agente debe incluir un bloque ## CLAUDE_TRAILER al final del handoff con campos como AGENT: developer, TASK_ID: ..., OUTCOME: ..., NEXT_STATUS: .... La marca AGENT: hace que el fallback distinga el trailer de esta ejecución del de un agente anterior en el mismo fichero acumulativo. Cubre el caso real "developer escribió 170 líneas de handoff pero el mensaje final se cortó antes del bloque trailer".

⑥ hook_session_context.py

SessionStart · INYECTA CONTEXTO

Cuando arrancas Claude Code o tras /clear, prepara el contexto del primer turno.

Inyecta como additionalContext

  • Phase y task activos + status
  • Spawn count vs budget
  • Último worker que tocó algo
  • Journeys pendientes (con bloqueo de /next-slice)
  • Doc discrepancies sin resolver
  • Sugerencias de mantenimiento (PROGRESS.md > 8KB, ledger > 200KB)
  • Errores recientes de hooks (de hook-errors.log)
  • Head compacto de PROGRESS.md (≤3000 chars)

Es la magia detrás del "resiliente a /clear": tras borrar contexto, el siguiente turn arranca con todo lo crítico pre-cargado.

04Manejo de errores en hooks

Si un hook lanza excepción, NO propaga: escribe a orchestrator-state/hook-errors.log y devuelve exit 0.

orchestrator-state/hook-errors.log

2026-05-13T10:23:14Z  hook_write_scope_guard.py  IOError  registry.json
                       Reading registry failed: [Errno 13] Permission denied
                       Tool=Write  Path=src/features/x.py  TASK=P02-S03-T001
2026-05-13T11:45:02Z  hook_capture_subagent_stop.py  KeyError  'OUTCOME'
                       Agent=validator  Trailer didn't include OUTCOME field

El hook SessionStart surface estos errores al inicio: si hay entradas recientes (<24h), las muestra como aviso.

05Locks POSIX: fcntl.flock

Validator y tester corren en paralelo y ambos quieren mutar el mismo registry.json. Sin lock, races silenciosas corrompen el estado.

El context manager file_lock

# En .claude/bin/common.py
@contextmanager
def file_lock(path, shared=False):
    """
    Adquiere fcntl.flock sobre <path>.lock.
    LOCK_EX (exclusivo) por defecto, LOCK_SH si shared=True.
    Reentrante por proceso vía contador _LOCK_DEPTH.
    """
    lock_path = path + ".lock"
    key = os.path.abspath(lock_path)

    if _LOCK_DEPTH.get(key, 0) > 0:
        _LOCK_DEPTH[key] += 1
        yield
        _LOCK_DEPTH[key] -= 1
        return

    fd = open(lock_path, "a+")
    try:
        fcntl.flock(fd, fcntl.LOCK_SH if shared else fcntl.LOCK_EX)
        _LOCK_DEPTH[key] = 1
        yield
    finally:
        _LOCK_DEPTH[key] = 0
        fcntl.flock(fd, fcntl.LOCK_UN)
        fd.close()

Propiedades clave

  • Exclusivo por defecto: otros procesos esperan a nivel kernel.
  • Reentrante por proceso: el mismo proceso puede entrar dos veces sin deadlock (contador _LOCK_DEPTH).
  • POSIX-only: en Windows es no-op.
  • Liberación garantizada: el finally libera incluso si hay excepción.

Escrituras atómicas

write_json y append_jsonl hacen temp file + os.replace bajo lock, con os.fsync. Un crash a media operación deja el fichero original íntegro.

06La carrera resuelta: validator ‖ tester

Coreografía completa cuando ambos paralelos terminan.

Validator (info-only) llega primero

t=0SubagentStop dispara para validator
t=1parsea trailer: OUTCOME=approved
t=2flock(registry) adquirido
t=3schema dice info_only=true
t=4task.validator_outcome = approved
t=5NO toca task.status
t=6flock liberado

Tester (lifecycle) llega segundo

t=2SubagentStop dispara para tester
t=3parsea trailer: OUTCOME=pass
t=4flock(registry) ocupado · espera
t=6flock adquirido tras liberación
t=7schema dice lifecycle=true
t=8task.status = ready_for_close
t=9flock liberado

Resultado final en registry

"P02-S03-T001": {
  "status": "ready_for_close",        # tester gana
  "validator_outcome": "approved",    # validator persisted
  "validator_next_status": "ready_for_close",
  "tester_outcome": "pass",
  ...
}

Si el orden fuera al revés (tester primero), el resultado sería idéntico: la regla "tester gana lifecycle, validator gana metadata" está en el schema, no depende del orden.

El truco: el schema dice quién muta qué. El hook lee el schema. Cambiar la naturaleza de un rol requiere editar orchestrator-contract.json → trailer_schema.roles.<rol>.info_only.

07Orden de locks (project-wide)

Cuando un hook o script toca varios ficheros bajo lock, el orden es siempre el mismo. Evita deadlocks.

1. file_lock(registry.json)         # siempre primero
2. file_lock(runtime-state.json)    # siempre segundo

Quién aplica este orden

  • hook_capture_subagent_stop.py
  • claim_task.py
  • register-followup-task.sh promote
  • remove_pending_journey_verification
  • waive_journey_verification
  • sync_runtime_state_from_registry

Si alguna función tomara runtime-state primero y luego registry, abriría ventana de deadlock con otra que ya tiene registry y quiere ir a runtime-state.

09Permisos y bypass

El orquestador corre con bypassPermissions. Los hooks aplican su propia capa por encima.

Capa Claude Code (permission modes)

Cuatro modos: default, acceptEdits, plan, bypassPermissions.

bypassPermissions salta prompts interactivos.

Capa del orquestador (hooks)

El bypass no afecta a los hooks. hook_write_scope_guard.py sigue denegando escrituras peligrosas. hook_spawn_budget.py sigue cortando el spawn 21.

Nunca uses bypassPermissions en exploración interactiva con prompts externos sin guardrails. Es seguro aquí porque los hooks son rigurosos.
