Máquina de estados completa

Esta página rompe el "happy path" lineal y enseña cada estado posible del orquestador: cómo se mueve una task por su lifecycle, qué transiciones disparan los trailers, cómo entra y sale del estado blocked, y qué pasa cuando varios subagentes terminan a la vez. Pensada para ingenieros que necesitan razonar sobre fallos, no sólo sobre el caso bueno.

01Qué máquinas de estado tiene el orquestador

El sistema combina seis máquinas de estado que comparten registro y locks. Conocer las 6 — y cómo se sincronizan — es lo que permite operar el orquestador en producción.

① Lifecycle de una Task

El estado canónico de cada TASK_ID en registry.json → tasks[].status. 9 estados, transiciones disparadas por trailers de subagentes bajo lock.

② Lifecycle de una Tool Call

Cada invocación de Write/Edit/Bash/Agent atraviesa PreToolUse → ejecución → PostToolUse. PreToolUse puede denegar; los demás son auditoría.

③ Lifecycle de un Subagente

Desde que main-orchestrator invoca Agent hasta que el SubagentStop hook parsea el trailer y muta el registry bajo lock.

④ Lifecycle de un Journey

Recorridos del usuario declarados en la matriz. Estados: open, pending_verify, verified, waived.

⑤ Lifecycle de un Follow-up

Trabajo descubierto fuera de scope. Estados: proposed, promoted, waived.

⑥ Lifecycle de una Phase

Bloque de tasks con gate humano antes de abrir la siguiente. not_started → in_progress → gate_pending → done.

02Máquina de estados de una Task

El estado de una task vive en registry.json → tasks["P02-S03-T001"].status. Sólo agentes lifecycle (developer, tester, slice-verifier, debugger, closer, deployer) pueden mutarlo; el hook SubagentStop es el único punto de mutación y lo hace bajo file_lock(registry.json).

Los 10 estados explicados

pending
Estado inicial tras bootstrap. Las deps no están todas done aún. No es reclamable.
ready
Todas las deps done. promote_ready_tasks la promovió. Reclamable.
claimed
Un terminal la tomó con claim_task.py bajo lock. Otros terminales no pueden reclamarla.
in_progress
El planner emitió OUTCOME: ready · CONTEXT_READY: yes. El developer empieza a editar.
validator_tester_pending
El developer terminó con éxito. validator‖tester van a correr en paralelo.
needs_debug
El tester falló o el validator pidió cambios → el debugger debe arreglar.
ready_for_close
Validator approved + tester pass. Listo para /verify-slice humano con MCP browser.
verified_pending_close
slice-verifier verificó con MCP, datos reales/proporcionados, logs y evidencia. Todavía no está done; falta closer/Git.
done
Closer commiteó, pusheó, baseline sincronizado, worktrees limpios. Las 5 pruebas *_READY emitidas.
blocked
Algo bloqueó terminalmente: 3 ciclos debugger fallidos, push falló, schema mismatch, conflicto con otra task activa. Requiere decisión humana.

04Trailer → mutación de estado (mapeo schema-aware)

El hook hook_capture_subagent_stop.py no tiene lógica hardcodeada sobre qué agente muta qué. Lee el schema de orchestrator-contract.json y aplica la regla.

def on_subagent_stop(trailer):
    role = trailer.role
    role_schema = contract.trailer_schema.roles[role]

    # 1. validar OUTCOME contra enum
    if trailer.OUTCOME not in role_schema.outcome_values:
        log_hook_error(f"invalid OUTCOME={trailer.OUTCOME} for {role}")
        return  # no muta

    # 2. validar NEXT_STATUS si el rol lo declara required
    if "NEXT_STATUS" in role_schema.required_keys:
        if trailer.NEXT_STATUS not in role_schema.next_status_values:
            log_hook_error(...)
            return

    # 3. adquirir locks en orden
    with file_lock(registry_path):
        with file_lock(runtime_state_path):
            if role_schema.info_only:
                # validator, official-docs-researcher, planner, ...
                task[f"{role}_outcome"] = trailer.OUTCOME
                task[f"{role}_next_status"] = trailer.NEXT_STATUS
                # NO muta task.status
            elif role_schema.mutates_registry_lifecycle:
                # developer, tester, slice-verifier, debugger, closer, deployer
                # Guardrail especial: closer needs 7 readys
                if role == "closer" and trailer.OUTCOME == "committed":
                    if not all_readys_present(trailer):
                        trailer.OUTCOME = "blocked"
                        trailer.NEXT_STATUS = "blocked"
                        log_hook_error("closer missing *_READY/RUNTIME_CLEANED proofs")

                task.status = trailer.NEXT_STATUS
                task[f"{role}_outcome"] = trailer.OUTCOME

            # 4. journeys y FU
            process_journey_trailer(trailer)
            process_followup_trailer(trailer)

            # 5. promote downstream
            promote_ready_tasks()

    save_registry()
    save_runtime_state()

06Lifecycle de un Subagente

Desde que el main-orchestrator lo invoca hasta que muta el registry. Este es el camino crítico — cada subagente lo recorre por completo.

Qué pasa si el trailer es inválido

  • OUTCOME fuera de enum: el hook escribe a hook-errors.log y NO muta lifecycle. La task se queda en su estado anterior.
  • NEXT_STATUS fuera de enum (cuando es required): mismo tratamiento, no muta.
  • TASK_ID distinto de CLAUDE_ACTIVE_TASK_ID (terminal worker): el hook lo etiqueta como scope mismatch, registra la nota en ledger, pero rechaza la mutación. Esto evita que un subagente desbocado mute la task del terminal vecino.
  • Schema ausente o rol no declarado: el hook escribe error visible y se niega a mutar (fail-safe).
  • [Nuevo] Trailer ausente (mensaje truncado): caso real "developer escribió 170 líneas de handoff pero su mensaje final se cortó antes del bloque CLAUDE_TRAILER:". Antes la task quedaba colgada en in_progress. Ahora el hook hace fallback al handoff en disco: lee orchestrator-state/tasks/handoffs/<TASK_ID>.md, busca un bloque marcado ## CLAUDE_TRAILER con campos AGENT: + TASK_ID: coincidentes, y muta el registry desde ahí. Logea recovered_from_handoff en hook-errors.log para que SessionStart lo surfacee.

07La carrera resuelta: validator ‖ tester (sequence diagram)

Dos subagentes terminan a la vez. Ambos quieren mutar el registry. El lock POSIX serializa la escritura; el schema decide quién gana lifecycle.

[validator] [tester] [Claude Code] [SubagentStop hook] [registry.json] │ │ │ │ │ │ emite trailer ────────────────────────────►│ │ │ │ │ emite trailer ─────►│ │ │ │ │ │ │ │ │ │ │── stop(validator) ──────►│ │ │ │ │── stop(tester) ─────────►│ │ │ │ │ │ (carrera, llegan ~iguales)│ │ │ │ │ │ │ │ │ │ flock(registry) [EX] ─────►│ adquirido por validator │ │ │ │ schema.role=validator │ │ │ │ │ info_only: true │ │ │ │ │ task.validator_outcome = │ │ │ │ │ approved │ │ │ │ │ save_registry() │ atómico (temp+rename) │ │ │ │ flock release │ │ │ │ │ │ │ │ │ │ tester estaba waiting... │ │ │ │ │ flock(registry) [EX] ─────►│ adquirido por tester │ │ │ │ schema.role=tester │ │ │ │ │ mutates_lifecycle: true │ │ │ │ │ task.status = │ │ │ │ │ ready_for_close │ │ │ │ │ save_registry() │ atómico │ │ │ │ flock release │ │ │ │ │ │ │ │ │ │ resultado deterministico: │ │ │ │ │ tester gana status, │ │ │ │ │ validator gana metadata │
Si llegaran al revés (tester primero), el resultado sería idéntico. La regla "tester gana lifecycle, validator gana metadata" está en el schema, no en el orden de llegada. El lock garantiza atomicidad, el schema garantiza determinismo.

08Máquina de estados de un Journey

Los journeys son recorridos del usuario declarados en la Journey Coverage Matrix. Su estado vive en registry.json → journeys[].verification_status.

Cómo afecta al DAG

Mientras un journey está en pending_verify, runtime-state.pending_journey_verifications[] contiene su JID. /next-wave y claim_task.py filtran (defieren) sólo las tasks que referencian ese JID. El resto del DAG sigue corriendo. En DAG-only no existe modo "strict" que bloquee todo el grafo.

09Máquina de estados de un Follow-up

Los Follow-ups (FU) son trabajo descubierto fuera del scope de la slice actual. Su estado vive en runtime-state.open_followups[] y en el fichero YAML follow-ups/FU-<id>.yaml.

FU bloqueantes (severity high|critical|blocker)

Mientras existe un FU proposed · high|critical|blocker, el orquestador bloquea:

  • /next-wave — no propone nuevas tasks.
  • claim_task.py — rechaza claims.
  • closer — devuelve blocked si encuentra FU bloqueantes asociadas a la slice.

Para desbloquear hay que /promote-followup (humano firma PROMOTE <ID>) o register-followup-task.sh waive.

10Máquina de estados de una Phase

Las phases son lotes de tasks que comparten contexto de producto. Hay un gate humano explícito antes de abrir la siguiente.

Qué valida /phase-gate <PHASE_ID>

  • Todas las tasks de la phase en status: done.
  • Todos los journeys de esa phase en verification_status: verified | waived.
  • Sin FU proposed con severity high|critical|blocker que referencien tasks de la phase.
  • (Opcional con --require-git-clean) árbol de trabajo limpio.
  • Existencia de handoff + evidence + report para cada task.

11Contador de spawn budget

No es una máquina de estados clásica, pero es un contador acotado que bloquea cuando se agota. Vive en runtime-state.json → spawn_counts[TASK_ID].

[main-orchestrator] [PreToolUse Agent hook] [runtime-state.json] │ │ │ │── Agent: planner ─────────►│ │ │ │ count = 0 → allow │ │ │ count = 1 ───────────────────►│ │ │ │ │── Agent: developer ───────►│ │ │ │ count = 1 → allow │ │ │ count = 2 ───────────────────►│ │ │ │ │ ... pasan 18 spawns ... │ │ │ │ │ │── Agent: slice-verifier ────►│ │ │ │ count = 19 → allow │ │ │ count = 20 ──────────────────►│ │ │ │ │── Agent: extra (21º) ──►│ │ │ │ count = 20 → DENY │ │ │ "Spawn budget agotado" │ │◄── permissionDecision: deny│ │ │ │ │ │ main-orchestrator no puede invocar más agentes en esta slice │ debe cerrar con lo que tiene o bloquear la task

El counter se incrementa al final del SubagentStop hook (cuando el subagente termina con éxito). Si el hook capture rechaza la mutación por trailer inválido, el counter no avanza.

13Lo que persiste vs lo que es efímero

Reglas de retención sobre /clear, /compact y restart del terminal.

Persiste (sobrevive a todo)

  • orchestrator-state/tasks/registry.json — estado canónico de tasks/journeys/phases
  • orchestrator-state/tasks/runtime-state.json — counters, último worker, pending journeys, FU abiertas
  • orchestrator-state/tasks/handoffs/<TASK_ID>.md — historia secuencial de la slice
  • orchestrator-state/tasks/evidence/<TASK_ID>/
  • orchestrator-state/tasks/reports/<TASK_ID>.md
  • orchestrator-state/memory/PROGRESS.md
  • orchestrator-state/agent-memory/<agente>/MEMORY.md
  • orchestrator-state/tasks/follow-ups/FU-*.yaml
  • orchestrator-state/hook-errors.log
  • Los 5 docs source-of-truth (Git-controlled)

Efímero (se pierde con /clear o restart)

  • El contexto conversacional del agente actual
  • Cualquier "lo que estaba pensando" del modelo
  • Variables no exportadas del terminal
  • CLAUDE_ACTIVE_TASK_ID si haces unset o abres terminal nuevo
  • Worktree dirty cambios sin commit (si no estaban en el commit del closer)

Recuperación tras /clear

  1. SessionStart hook dispara automáticamente.
  2. Inyecta additionalContext con: phase/task activos, spawn count, journeys pendientes, doc discrepancies, head de PROGRESS.md, errores recientes.
  3. El siguiente turn del agente ya tiene lo crítico cargado.
  4. Para tareas concretas (/verify-slice, /next-slice), el comando reconstruye lo que necesita desde disco con reads explícitos.

14Sincronización entre las 6 máquinas

Las 6 máquinas no son independientes. Una transición en una puede disparar transiciones en otras. Aquí están los acoples.

OrigenDisparoEfecto en otra máquina
Task: → doneSubagentStop con closer commit OKJourney: si era la última task, → all_tasks_done o verified (inline)
Task: → doneSubagentStopOtras tasks: promote_ready_tasks mueve sucesores pending → ready
Task: → doneSubagentStopPhase: si era la última task de la phase, → gate_pending
Journey: → pending_verifycloser emite JOURNEY_PENDING_VERIFYTasks que referencian el JID: claim_task.py las rechaza hasta verified o waived
FU: → proposed · high+register-followup-task.sh proposeTasks: /next-wave bloquea claims globalmente hasta resolución
FU: → promoted/promote-followupTasks: nueva fila en Coverage Registry + DAG regenerado + posible blocked: conflict_with_worker_task si choca
Phase: → done/phase-gatePhase N+1: → in_progress (puede empezar a reclamar tasks)
Tool call: deny PreToolUsespawn budget agotadoSubagente: nunca entra en running; main-orchestrator debe cerrar slice con lo que tiene
Toda esta sincronización ocurre dentro de file_lock(registry.json) en el SubagentStop hook. Por eso el orden de locks (registry primero, runtime-state después) es crítico: garantiza que las propagaciones cross-machine son atómicas.

15Resumen mental para ingenieros

Quien muta el estado

  • Task lifecycle → SubagentStop hook (lifecycle agents)
  • Journey → SubagentStop hook (closer/journey verify)
  • FU → register-followup-task.py + /promote-followup
  • Phase → check_phase_gate.py (via /phase-gate)
  • Tool call → Claude Code runtime (con hooks)
  • Spawn count → SubagentStop hook (incremento) + PreToolUse Agent (read)

Quien NO puede mutar nada

  • Validator (info-only)
  • official-docs-researcher (info-only)
  • Planner (info-only para registry)
  • screen-journey-reviewer (info-only)
  • main-orchestrator (info-only, sólo orquesta)
  • Cualquiera con escritura directa al registry/runtime-state. Sólo scripts/hooks bajo lock.
Si tienes que debug un fallo de orquestador, mira primero:
  1. orchestrator-state/hook-errors.log — qué hook falló y por qué
  2. orchestrator-state/tasks/registry.json — estado real de la task
  3. orchestrator-state/tasks/runtime-state.json — spawn counts, pending journeys, open FU
  4. orchestrator-state/tasks/handoffs/<TASK_ID>.md — qué pasó en cada paso
  5. orchestrator-state/tasks/ledger.jsonl — qué writes/edits ocurrieron y cuándo
