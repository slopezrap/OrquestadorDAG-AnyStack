El orquestador, explicado de tripas a piel

Esta es la guía técnica completa de tu orquestador OrquestadorDAG-AnyStack: qué hace, cómo está construido, por qué cada pieza existe y cómo se coordinan entre sí. Pensada para ingenieros sin experiencia previa con IA agéntica.

Si lees esta página entera entiendes qué hace. Las otras 6 páginas son las tripas.

01¿Qué es esto, en una frase?

Es una línea de montaje automatizada que usa Claude (modelo de lenguaje de Anthropic) para construir aplicaciones fullstack de calidad de producción, paso a paso, con verificación humana en puntos clave y trazabilidad total en disco.

Piénsalo así: en una fábrica de coches, cada estación hace una sola cosa (chasis, motor, pintura, control de calidad). Aquí cada estación es un subagente de Claude con un rol acotado: uno planifica, otro programa, otro valida arquitectura, otro corre tests, otro arregla bugs, otro commitea a Git. El "orquestador" es quien decide qué estación entra en juego, en qué orden y bajo qué reglas.

Lo que NO es

  • No es un "ChatGPT que escribe código". Eso sería el modelo crudo. Aquí hay protocolo, gates, locks, hooks y disco alrededor del modelo para que se pueda confiar en su salida.
  • No es un prototipo: el contrato es código de producción desde el día 1 — validaciones reales, tests reales contra base de datos real, seguridad real.
  • No improvisa: cada cambio sigue un grafo dirigido (DAG) construido a partir de cinco documentos canónicos. Si los documentos no están bien, el sistema se niega a empezar.

02Modelo mental en 7 piezas

El orquestador es la suma de estas siete piezas. Cada una vive en una carpeta distinta del repo, y cada una tiene su propia página detallada en esta guía.

① Fuentes de verdad (5 docs)

Cinco documentos canónicos en docs/source-of-truth/ describen el producto, la arquitectura, las pantallas, el stack y el plan. Son la única autoridad: el orquestador deriva todo desde ahí.

② DAG explícito

Una tabla (Coverage Registry) en el checklist declara qué tareas dependen de cuáles. El bootstrap convierte esa tabla en un grafo (task-dag.json) que ordena el trabajo y permite paralelismo seguro.

③ Subagentes especializados

14 subagentes con un rol estricto cada uno: planner, developer, validator, tester, debugger, closer, etc. Cada uno declara qué puede escribir y qué no.

④ Hooks de Claude Code

Cuatro guardias que Claude Code dispara antes/después de cada herramienta: presupuesto de spawns, scope de escrituras, captura de resultados y contexto de sesión.

⑤ Locks POSIX para concurrencia

Cuando dos agentes (validator ‖ tester) terminan a la vez y quieren tocar el mismo fichero, un fcntl.flock serializa la escritura. Sin esto, el estado del proyecto se corrompería.

⑥ Scripts y CLI

Scripts en scripts/ y .claude/bin/ implementan las operaciones mecánicas: bootstrap, claim de tareas, checks de integridad, generación de contratos API, git workflow, follow-ups formales.

⑦ Memoria en disco

Toda la coordinación vive en orchestrator-state/: registry, runtime-state, handoffs, evidence, ledger, memoria por agente. Sobrevive a /clear y a reinicios. El disco es la verdad, el contexto es efímero.

03Los 5 documentos source-of-truth

Todo arranca aquí. Los cinco documentos viven en docs/source-of-truth/ y los rellena un humano (típicamente con ayuda de ChatGPT y los prompts de docs/prompts/). Sin ellos completos, bootstrap_source_of_truth.py rechaza arrancar el orquestador.

① instrucciones.md

Objetivo del producto, alcance, reglas de negocio y la Journey Coverage Matrix (recorridos del usuario).

② *_TECHNICAL_GUIDE.md

Arquitectura, contratos front↔back, endpoints, esquema de DB, ADRs, verification data contract.

③ *_IMPLEMENTATION_CHECKLIST.md

Fases, steps, Coverage Registry con dependencias, write sets, conflict groups, journey refs.

④ STACK_PROFILE.yaml

Stack concreto, paths del frontend, comandos de test/build, enforcer de tokens visuales, workflow Git.

⑤ UX_CONTRACT.md

Personas, inventario de pantallas, estados UI obligatorios (loading/empty/error/...) y reglas de verificación.

Por qué cinco y no uno: el orquestador es AnyStack: no asume ningún framework concreto. Para que el motor funcione con Flutter, Next.js, SwiftUI o lo que toque, el stack vive aparte (STACK_PROFILE.yaml) y los hooks/scripts despachan a plugins (.claude/enforcers/, .claude/git-workflows/) según ese perfil.

04El flujo en una imagen

Una slice es la unidad mínima de trabajo: una fila del Coverage Registry. Cada slice atraviesa esta cadena de subagentes. El budget total por slice es de 20 spawns máximo (lo enforce un hook).

ChatGPT rellena templates │ ▼ 5 docs source-of-truth ─────► bootstrap_source_of_truth.py │ ▼ registry.json + task-dag.json + work-items/*.yaml │ ▼ ┌────────── /next-wave (propone TASK_IDs ready y seguros) │ terminal worker export CLAUDE_ACTIVE_TASK_ID=P02-S03-T001 export CLAUDE_TASK_PACK=orchestrator-state/tasks/task-packs/P02-S03-T001.md │ ▼ /next-slice TASK_ID │ ├─ 1. planner (blocking · selecciona y prepara contexto) ├─ 2. developer ‖ official-docs-researcher (paralelo) ├─ 3. validator ‖ tester (paralelo) ├─ 4. debugger (sólo si tester=fail o validator=changes) └─ termina en "tester pass" │ ▼ /verify-slice ─ gate humano: hard reset + datos reales + reproducción │ ┌───────┴────────┐ │ verified │ issues_found ▼ ▼ closer debugger ◄── vuelve al paso 3 (commit atómico + push + cleanup)

05Las 4 ideas clave

Si solo recuerdas cuatro cosas de este orquestador, recuerda éstas.

Cambios recientes integrados: pr-flow es ahora el git_workflow por defecto (cada terminal en su worktree, PRs auto-mergeados sin tu validación), el bloque copy/paste de /next-wave hace unset de las 4 variables de scope antes de exportar las nuevas, el closer recupera el trailer del handoff si el mensaje del agente se trunca, scripts/dev-restart.profile.sh está protegido contra aplastamiento durante slice, y el contenedor <repo>-worktrees/ se borra solo cuando queda vacío. Detalles en las páginas de Hooks & Locks y Scripts & Git.
①

Una cosa a la vez, declarada

Cada terminal tiene exactamente un CLAUDE_ACTIVE_TASK_ID activo. Sin tarea explícita no se trabaja. El DAG explícito impide que dos terminales pisen los mismos ficheros.

②

Disco > contexto

Cada agente lee y escribe estado en ficheros (handoffs/, evidence/, registry.json). Cuando haces /clear, el contexto se borra pero el trabajo persiste. /verify-slice reconstruye todo desde disco.

③

Gates humanos no opcionales

Ningún commit ocurre sin que un humano haya ejecutado /verify-slice con datos reales y reproducción en navegador. Un hook se niega a marcar la tarea como done sin esa prueba.

④

Reglas mecánicas, no culturales

Las reglas las imponen los hooks de Claude Code, no la educación del modelo: presupuesto de spawns, scope de escrituras, validación de trailer. Si el modelo intenta algo prohibido, el hook lo bloquea físicamente.

06Glosario operativo

Los términos que vas a oír en cada página. Si te pierdes en alguno, vuelve aquí.

Subagente
Una "instancia" de Claude con prompt + permisos propios, definida en .claude/agents/<nombre>.md. Se invoca desde el agente principal (main thread) con la herramienta Agent. Vive en su propio contexto: no contamina al resto.
main-orchestrator
El subagente que actúa como sesión principal de Claude Code. Es el único autorizado a orquestar al resto. Arranca con claude --agent main-orchestrator --permission-mode bypassPermissions.
Slice / Task
Una fila del Coverage Registry. Tiene TASK_ID, dependencias, write set, conflict group, acceptance, verify. Se ejecuta en un terminal único de principio a fin.
Pipeline per-slice
La cadena planner → developer → validator‖tester → debugger → verify-slice → closer. Máximo 20 spawns por slice (un hook lo enforce).
Handoff
El fichero orchestrator-state/tasks/handoffs/<TASK_ID>.md. Lo inicializa el developer y los siguientes agentes le añaden secciones. No se sobrescribe: solo se apendea.
Trailer
Las últimas líneas que cada subagente emite al terminar: TASK_ID:, OUTCOME:, NEXT_STATUS:, etc. El hook SubagentStop las parsea y actualiza el registry bajo lock.
Hook
Un script que Claude Code dispara automáticamente: antes de una tool (PreToolUse), después (PostToolUse), al terminar un subagente (SubagentStop), al iniciar sesión (SessionStart). Sólo PreToolUse puede bloquear.
DAG
Directed Acyclic Graph. Grafo dirigido sin ciclos de dependencias entre TASK_IDs, derivado de la columna Depends on del Coverage Registry.
Write set
Los globs de paths que una tarea está autorizada a tocar. Si una task escribe fuera de su write set, el hook avisa o bloquea.
Conflict group
Una etiqueta que comparten tareas que no pueden correr en paralelo aunque sus dependencias estén satisfechas.
Worktree aislado
Git crea una copia física separada del repo para que developer / debugger / deployer trabajen sin pisar el principal.
Journey
Un recorrido completo del usuario (multi-pantalla) declarado en la Journey Coverage Matrix de instrucciones.md.
Follow-up (FU)
Una propuesta formal de trabajo descubierto fuera del scope de la slice actual. Se crea con register-followup-task.sh propose.
Verification Data Contract
Datos reales o proporcionados (no mocks) que /verify-slice y /verify-journey usan para reproducir el flujo.
