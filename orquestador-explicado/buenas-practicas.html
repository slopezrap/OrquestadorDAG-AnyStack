Cómo pensar un orquestador (según Claude oficial)

Si mañana te pidieran construir un orquestador parecido para otro dominio (legal, médico, financiero), ¿cómo lo pensarías? Esta página resume cómo Claude Code recomienda diseñarlo y cómo cada decisión está implementada en este proyecto.

02Las restricciones que te impone Claude Code

Antes de diseñar nada, hay 4 reglas duras del runtime que tienes que aceptar.

① Un subagente no puede orquestar a otros subagentes

Sólo el agente del main thread puede invocar la herramienta Agent. Si un subagente intenta lanzar otros, Claude Code falla la operación. Previene cadenas infinitas y mantiene jerarquía de mando clara.

En este proyecto: el main-orchestrator arranca como sesión principal ("agent": "main-orchestrator"). Todos los demás se invocan desde él. NO lleva tools: en frontmatter para no convertirla en allowlist.

② Sólo PreToolUse puede bloquear una acción

De los 4 tipos de hook, sólo PreToolUse puede devolver permissionDecision: deny. PostToolUse, SubagentStop y SessionStart son informativos.

Aquí: los enforcement reales son PreToolUse: hook_spawn_budget.py y hook_write_scope_guard.py. El resto (ledger, capture, session context) son PostToolUse / SubagentStop / SessionStart.

③ El contexto del subagente es aislado

Cada subagente arranca con su propio system prompt y NO ve la conversación del main thread. Para pasarle información: prompt explícito o disco.

Aquí: los task-packs y handoffs son disco-first. El main-orchestrator pasa al developer el path al task pack; el developer lo lee y trabaja desde ahí.

④ Worktrees aislados son declarativos · pero session-level ≫ subagent-level

Claude Code permite declarar isolation: worktree en el frontmatter de un subagente. Sin embargo, si varios subagentes de la misma slice abren worktrees distintos, dejan de ver el trabajo del otro (validator no encuentra el código del developer, etc.).

Aquí: ningún subagente lleva isolation: worktree. El aislamiento Git por TASK_ID se hace una vez por terminal worker, antes de arrancar Claude Code, vía ./scripts/ensure-task-worktree.sh. Así todos los subagentes de la slice (developer, validator, tester, debugger, closer) inspeccionan y editan el mismo checkout. El paralelismo se mantiene a nivel terminal: dos terminales en dos TASK_IDs distintos → dos worktrees distintos.

03Los 10 principios para orquestadores robustos

Recogidos de la documentación oficial de Claude Code, traducidos y contextualizados.

1 · Separación de responsabilidades estricta

Cada subagente hace una sola cosa. Un agente "todo-en-uno" infla el contexto y genera decisiones inconsistentes.

Aquí: 13 agentes con scope acotado. planner sólo planifica, developer sólo programa, validator sólo revisa, tester sólo testea.

2 · Disco > contexto

El contexto de Claude es efímero (se borra con /clear). Los ficheros en disco son la única fuente persistente.

Aquí: todo el estado vive en orchestrator-state/ y los 5 docs source-of-truth. /verify-slice reconstruye todo desde disco. Puedes hacer /clear entre tester pass y verify sin perder nada.

3 · Trailers machine-readable

La conversación es texto libre, pero la coordinación entre agentes necesita campos parseables. Define un schema de "trailer" (líneas CLAVE: valor) que cada agente debe emitir. Un hook lo parsea.

Aquí: orchestrator-contract.json → trailer_schema declara required_keys, outcome_values, next_status_values por rol. hook_capture_subagent_stop.py los valida.

4 · Locks para concurrencia

Si dos subagentes pueden tocar el mismo fichero a la vez, envuelve la operación en fcntl.flock. Sin esto, races corrompen el estado.

Aquí: common.py → file_lock reentrante por proceso. Orden global: registry primero, runtime-state después.

5 · Dependencias explícitas, no implícitas

No le digas al modelo "después de A, haz B". Declara las dependencias en datos. El scheduler las consulta. Orden deterministico y paralelización segura.

Aquí: columna Depends on del Coverage Registry. bootstrap_source_of_truth.py deriva el grafo. /next-wave lo lee. El modelo nunca decide el orden.

6 · Gates con evidencia tangible

Antes de cualquier acción "no reversible" (commit, deploy), exige evidencia en disco. Screenshots, test outputs, curls, logs.

Aquí: el closer no commitea sin las 5 pruebas *_READY=yes. check-handoff-contract.sh rechaza handoffs sin secciones machine-readable. /verify-slice exige reproducción humana con datos reales.

7 · Permisos en capas (defense in depth)

Combina los permisos de Claude Code (permissionMode) con guardrails propios (hooks). Si el modelo decide algo peligroso, queda atrapado en al menos una capa.

Aquí: el orquestador corre con bypassPermissions (sin prompts), pero los hooks PreToolUse aplican scope guard, spawn budget y discrepancy check.

8 · Memoria persistente por agente

Cada subagente puede mantener su memoria a largo plazo en disco (MEMORY.md). Sobrevive a /clear.

Aquí: orchestrator-state/agent-memory/<agente>/MEMORY.md. compact-agent-memory.py la comprime sin perder info (archiva original primero).

9 · Reseteo estratégico de contexto

El contexto crece. Entre fases críticas, ejecuta /clear. Los artefactos en disco hacen que el siguiente turn reconstruya lo que necesita.

Aquí: el flujo: /next-slice hasta tester pass → /clear → /verify-slice. hook_session_context.py reinyecta lo crítico tras el clear.

10 · Reglas mecánicas, no culturales

"El developer debe respetar el write set" es cultura — el modelo puede olvidarlo. "El hook bloquea cualquier write fuera del write set" es mecánica — no se puede saltar.

Aquí: spawn budget (hook), write scope guard (hook), closer guardrail (hook), git-workflow rechazando stash (script), trailer schema (validación machine), claim atómico (lock+check).

04Cómo diseñarías uno desde cero (10 pasos)

1

Define las fuentes de verdad

¿Cuáles son los documentos canónicos que describen el dominio? Aquí son 5. En otro proyecto podrían ser 3 o 7. Decide qué NO es source-of-truth: todo lo demás es derivado.

2

Identifica los roles necesarios

Lista las responsabilidades: planificar, implementar, revisar, testear, arreglar, cerrar. Cada una → un subagente. Si dudas en separar dos roles, sepáralos.

3

Diseña el trailer schema

Decide qué OUTCOME puede emitir cada agente y qué NEXT_STATUS. Codifícalo en JSON. Si un agente muta lifecycle, marca info_only: false. Resuelve race conditions de pares paralelos.

4

Modela las dependencias

Si el trabajo es secuencial, lista. Si quieres paralelismo, grafo. Decide qué columna del documento canónico es la fuente del grafo y materialízalo en runtime.

5

Define write sets y conflict groups

Para cada tarea: ¿qué ficheros toca? ¿Qué tareas no deben correr a la vez? Sin esto no puedes paralelizar con seguridad.

6

Diseña los hooks mínimos

Los 4 imprescindibles:

  1. Write scope guard (PreToolUse)
  2. Trailer capture (SubagentStop)
  3. Spawn budget (PreToolUse Agent)
  4. Session context (SessionStart)
7

Elige el modo de permisos

bypassPermissions sólo si tienes hooks rigurosos. Para exploración o prototipos, mejor acceptEdits.

8

Decide cómo aislar el Git por tarea

Dos opciones reales:

  • Subagent-level (isolation: worktree en frontmatter): cada subagente abre su propio worktree. Útil si cada agente es totalmente independiente.
  • Session-level (recomendado para pipelines colaborativos): el terminal worker entra en el worktree del TASK_ID antes de arrancar Claude Code. Todos los subagentes de la sesión comparten checkout. Es lo que hace este proyecto vía ensure-task-worktree.sh.

Para orquestadores con handoff (developer → validator → tester → closer), session-level evita el caso "cada agente ve un árbol distinto".

9

Diseña los slash commands

Tres canónicos: /next-wave (read-only), /next-slice (ejecutar), /verify-slice (gate humano). En tu dominio podrían ser otros, pero piensa siempre en pares "ejecutar / verificar".

10

Establece el cierre Git como ritual

Define: qué puede estar dirty, qué jamás, quién hace el push, cómo se prueba que fue exitoso (las 5 marcas *_READY=yes).

05Anti-patrones reconocidos

Subagentes anidados

Diseñar un subagente "supervisor" que invoca otros. Claude Code no lo permite. Solución: ese supervisor es el main-orchestrator.

Estado solo en contexto

Confiar en el historial conversacional. Se pierde con /clear. Solución: handoffs en disco con secciones claras.

Scope creep en debugger

Un debugger que añade pantallas/endpoints para arreglar un test. Solución: debugger sólo toca paths del write_set. Si necesita más, FU formal.

git stash como flujo normal

Esconder cambios durante el cierre. Genera estados sin historia. Solución: closer commitea atómico. Cualquier cambio fuera del commit es FU.

tools: allowlist en main-orchestrator

Listar tools en frontmatter parece prudente pero convierte la lista en allowlist. MCPs nuevos no llegan. Solución: omitir tools:.

Mocks en lugar de datos reales

Verificar con datos inventados que pasan tests pero no representan producción. Solución: Verification Data Contract. Sin datos reales → FU formal.

06Patrones recomendados

Patrón: separación clara entre "ejecutar" y "verificar"

El pipeline puede ser automático; la verificación es donde el humano (o un check determinístico) decide. Aquí: /next-slice ejecuta. /verify-slice es el gate humano. El cierre lo invoca /verify-slice, nunca /next-slice.

Patrón: skills para checks reutilizables, subagentes para roles

Un skill es un fragmento que el modelo invoca cuando detecta un trigger. Un subagente es un rol con contexto propio. Aquí: skills (dev-loop, dev-verify, write-handoff, close-task) son rutinas que cualquier agente puede invocar. Los 14 subagentes son los roles.

Patrón: trailers como API contract entre agentes

Si pides "resumen estructurado", el modelo a veces inventa campos. Si declaras un schema y un hook que lo valida, el output queda forzado. Aquí: trailer_schema.roles.<agente> declara campos required y enums válidos.

Patrón: ledger inmutable como auditoría

Cada modificación queda registrada en ledger append-only. Aquí: ledger.jsonl versionado + bash-ledger.jsonl no versionado (para no re-ensuciar post-push).

Patrón: configuración del stack en datos, no en código

Un orquestador AnyStack no debe asumir framework. Solución: STACK_PROFILE.yaml declarativo que los scripts leen. Aquí: framework, paths, comandos, enforcer visual, workflow Git. Scripts delegan a plugins según el profile.

Patrón: comandos slash idempotentes y resilientes a /clear

Si tus comandos slash reconstruyen estado desde disco, sobreviven a /clear. Aquí: /verify-slice lee PROGRESS.md + registry + runtime-state + handoff + task pack + technical guide. No depende de contexto previo.

07Plantilla mental para nuevos orquestadores

┌─────────────────────────────────────────────────────────────────────┐ │ PLANTILLA DE ORQUESTADOR CLAUDE CODE │ ├─────────────────────────────────────────────────────────────────────┤ │ ① Fuentes de verdad (N docs declarativos) │ │ → describen el dominio, los reviewa un humano │ │ ② Bootstrap → estado runtime derivado │ │ → registry.json + grafo de dependencias + work items │ │ ③ Subagentes con UN rol claro cada uno │ │ → frontmatter mínimo, isolation: worktree sólo si edita código │ │ ④ Trailer schema machine-readable │ │ → orchestrator-contract.json con outcomes, next_status, info_only│ │ ⑤ Hooks como guardrails mecánicos │ │ → write scope, spawn budget, trailer capture, session context │ │ ⑥ Locks POSIX para concurrencia │ │ → file_lock(state) en orden global consistente │ │ ⑦ Comandos slash de entrada │ │ → /next-wave (ver) · /next-slice (ejecutar) · /verify-slice (gate)│ │ ⑧ Cierre con evidencia tangible │ │ → 5 marcas *_READY=yes, commit atómico, push vía workflow plugin│ └─────────────────────────────────────────────────────────────────────┘

Adaptas las piezas al dominio. Ejemplo para revisión legal de contratos:

  • Source-of-truth: el contrato, la cláusula estándar, la matriz de riesgos.
  • Subagentes: extractor, validator, annotator, summarizer.
  • Trailer: OUTCOME: aligned | divergent | missing; SEVERITY: low | medium | high.
  • Gate humano: revisión por el abogado antes de devolver al cliente.

La estructura es la misma. Cambia el dominio.

09Cuándo NO usar este patrón

✓ Vale la pena

  • Estás construyendo producción
  • Múltiples agentes paralelos tocan estado compartido
  • Necesitas auditoría / trazabilidad legal
  • El proyecto durará semanas/meses
  • Hay varios humanos con roles distintos
  • Las decisiones erróneas son costosas de revertir

✗ Es overkill

  • Una sola persona, un script puntual
  • POC / prototipo throw-away
  • Tareas que duran < 1 hora
  • Sin estado compartido entre pasos
  • Sin gates humanos necesarios
  • El coste de fallo es trivial
