Los 14 subagentes

Un subagente es Claude con un prompt acotado, una lista de herramientas y, opcionalmente, un worktree aislado. Cada uno hace una sola cosa bien. Aquí están los 14 que componen tu orquestador, qué escriben y qué emiten al terminar.

01Qué es exactamente un subagente

Cuando Claude Code arranca, hay un agente principal (el main thread). Ese agente puede invocar a otros agentes con la herramienta Agent. Cada uno se define en un fichero markdown en .claude/agents/<nombre>.md.

El fichero de un subagente

Cada agente es un .md con dos partes: frontmatter YAML (configuración) y cuerpo markdown (su system prompt).

---
name: developer
description: Implementa código del slice activo
model: sonnet
maxTurns: 150
permissionMode: bypassPermissions
---

# Developer agent

You are the developer. Your job is to implement
the code for the current TASK_ID exactly as the
task pack describes...

## Task worktree contract

This agent does not request its own nested
`isolation: worktree`. `/next-wave` launches
the whole Claude Code worker session inside
the per-TASK_ID worktree when the Git workflow
uses feature branches/PRs. All subagents in
the slice must inspect and edit that same
checkout.

Cómo se invocan

El agente principal escribe un mensaje Agent con el nombre del subagente y un prompt. Claude Code:

  1. Crea una sesión nueva con el prompt del subagente.
  2. El subagente hereda el cwd del terminal worker (que ya es el worktree del TASK_ID en pr-flow).
  3. Cuando el subagente termina, devuelve un mensaje final con un trailer.
  4. El hook SubagentStop parsea ese trailer y actualiza disco.
Regla dura: un subagente no puede invocar a otro. Solo el main thread orquesta. Esto es una restricción de Claude Code, no del proyecto.
Cambio reciente · worktree a nivel sesión: ningún subagente lleva ya isolation: worktree en su frontmatter. El aislamiento Git por TASK_ID ahora se hace antes de arrancar Claude Code, vía ./scripts/ensure-task-worktree.sh. Así todos los subagentes de la slice ven el mismo checkout (developer escribe, validator/tester lo inspeccionan en el mismo árbol) en vez de cada uno crear un worktree separado.

04El trailer schema

Cuando un subagente termina, emite un bloque final de líneas CLAVE: valor. El hook SubagentStop las parsea. Los valores válidos están declarados en .claude/orchestrator-contract.json → trailer_schema.roles.

Claves globales

TASK_ID:                 # coincide con CLAUDE_ACTIVE_TASK_ID
OUTCOME:                 # varía por rol
NEXT_STATUS:             # sólo roles que mutan lifecycle
HANDOFF:                 # path al handoff actualizado
EVIDENCE:                # path al directorio de evidencia
REPORT:                  # path del report (sólo closer)
REPORT_READY: yes|no     # closer
BASELINE_SYNC_READY:     # closer
GIT_READY:               # closer
PUSH_READY:              # closer
WORKTREES_CLEANED:       # closer
JOURNEY_PENDING_VERIFY:  # JID si cierra journey aparte
JOURNEY_VERIFIED_INLINE: # JID si verificado inline
JOURNEY_VERIFY_OUTCOME:  # verified | issues_found

05Fichas del pipeline per-slice

planner opus · maxTurns 80

blockinginfo-onlymemoria

Primer agente del pipeline. Su misión: dado un TASK_ID, dejar el contexto preparado para el resto sin volver a leer los 5 source-of-truth completos.

Qué lee
  • PROGRESS.md
  • registry.json (su TASK_ID)
  • runtime-state.json
  • Las secciones relevantes de los 5 docs
  • Su MEMORY.md
Qué escribe
  • task-packs/<TASK_ID>.md con front→back→DB map, allowed_paths, acceptance, verify, journey refs
  • Apuntes en su MEMORY.md
Gates que comprueba
  • Deps del TASK_ID = done
  • Sin journeys pendientes
  • Sin FU bloqueantes
  • Spawn budget no agotado
Trailer típico
OUTCOME: ready
CONTEXT_READY: yes
NEEDS_OFFICIAL_DOCS: yes|no
CLOSES_JOURNEY: J104|none

Si emite CONTEXT_READY: no → todo el pipeline se detiene.

developer sonnet · maxTurns 150

lifecycleworktree-sesiónmemoria

El único que escribe código de producto. Trabaja en el checkout que ya tiene el terminal worker (en pr-flow, el worktree dev/<TASK_ID> creado por ensure-task-worktree.sh; en push-to-main, directamente main). Sigue orden estricto: DB → backend → frontend.

Qué hace en orden
  1. Migración + entidad DB
  2. Repository + service backend
  3. Endpoint + tests backend
  4. Use case + data layer frontend
  5. UI + tests frontend
  6. Actualiza PROGRESS.md
  7. Inicializa handoff
Trailer
OUTCOME: success
NEXT_STATUS: validator_tester_pending
HANDOFF: ...
EVIDENCE: ...

Logging obligatorio: cada función emite log BEFORE/AFTER. Verbosidad controlada por ENABLE_VERBOSE_LOGGING; validator chequea ambos modos.

official-docs-researcher sonnet · maxTurns 50

info-onlyparalelo

Sólo si planner marcó NEEDS_OFFICIAL_DOCS: yes o la slice toca librerías volátiles. Corre en paralelo con developer en el mismo mensaje.

Cuándo
  • AI/ML volátiles (LangChain, OpenAI SDK, ...)
  • SDKs en major bump
  • APIs externas nuevas
  • Patrones de seguridad / streaming / OAuth
Qué produce
  • Notas en memory/official-doc-notes/*.md
  • Si discrepancia, el hook docs-discrepancy alerta al developer en el siguiente Write (warn-only)

validator opus · maxTurns 60

info-onlyparalelo con tester

Revisa cómo está hecho el código. No corre tests, no toca código. Lee diff + handoff + contexto.

Qué revisa
  • Clean Architecture
  • File size ~200/300 · funciones <50
  • Logging BEFORE/AFTER + ambos modos verbose
  • Tests reales (no mocks de negocio)
  • PROGRESS.md actualizado
  • Security checklist
  • A11y
  • Journey matrix / wiring drift
Por qué info-only

Corre en paralelo con tester. Para evitar race en el registry, tester es dueño del status; validator escribe validator_outcome y validator_next_status como metadatos.

Pero changes_requested bloquea al closer (que lee el handoff).

tester sonnet · maxTurns 80

lifecycleparalelo con validator

Tests reales contra backend + DB + frontend levantados. Backend tests pegan a DB real, frontend integración hace requests reales al backend.

Qué ejecuta
  • Backend lint + tests
  • Frontend lint + tests
  • Smoke curl
  • Logs con ambos modos ENABLE_VERBOSE_LOGGING
  • Outputs en evidence/<TASK_ID>/
Trailer
OUTCOME: pass | fail
NEXT_STATUS: ready_for_close
             | needs_debug
EVIDENCE: ...

Si fail → debugger. Máximo 3 ciclos; al 4º se bloquea con max_debug_cycles_reached.

slice-verifier sonnet · maxTurns 130

lifecycleMCP browserhandoff

Gate humano-real previo al closer. Hace hard reset, carga datos reales/proporcionados, reproduce la app con Chrome DevTools MCP, claude-in-chrome o Agent360 Browser MCP (`browser-mcp`), observa logs front/back/DB y apendea ## verify-slice. El aumento a maxTurns 130 sólo da margen al MCP browser; no cambia los 20 spawns por slice.

Estado: verified → verified_pending_close. Si no hay MCP browser conectado, bloquea con browser_mcp_unavailable; no hace fallback API-only.

debugger opus · maxTurns 120

lifecycleworktree-sesiónmemoria

Fix mínimo dentro del mismo write_set, en el mismo checkout que el developer (no abre un worktree nuevo). NO cambia scope, NO toca otras tasks, NO añade features.

Reglas
  • Máx 3 ciclos por TASK_ID
  • 4º intento → blocked · max_debug_cycles_reached
  • Si fix sale del write_set → no fix, sino FU formal
Trailer
OUTCOME: fixed | blocked | failed
NEXT_STATUS: validator_tester_pending
             | blocked

Tras el fix vuelve al paso 3 (validator ‖ tester).

screen-journey-reviewer sonnet · maxTurns 40

info-onlysólo verify-slice

Solo entra en /verify-slice cuando la slice es de pantalla, UX, journey o tiene VISUAL_CONTRACT_CHECK.

Estados UI que exige
  • loading
  • empty
  • error_network
  • error_validation
  • permission_denied
  • success
  • streaming / uploading / indexing (si aplica)
Política de FU

Único reviewer info-only que puede marcar followup_candidate=yes. /verify-slice decide si crear FU formal. El reviewer no la crea por sí mismo.

closer sonnet · maxTurns 50

lifecyclecierre con guardrail

El único que cierra una task. Lo invoca /verify-slice sólo cuando el router confirma verified_pending_close y un handoff ## verify-slice completo con MCP/datos/evidencia. Hace 6 cosas en orden:

  1. Pre-check del handoff con check-handoff-contract.sh.
  2. Escribe reports/<TASK_ID>.md.
  3. Sincroniza baseline con sync-product-baseline.sh sync.
  4. Pre-check Git: ./scripts/ensure-task-worktree.sh --check-current <TASK_ID>. En pr-flow exige el worktree/rama del TASK_ID; en push-to-main exige main.
  5. Commit atómico en el checkout actual del TASK_ID (main en push-to-main, dev/<TASK_ID> en pr-flow) · sin Co-authored-by · sin git stash.
  6. Push con ./scripts/git-workflow.sh.
  7. Housekeeping: slice-clean.sh + cleanup-worktrees.sh.
Cierre real sólo con las 5 pruebas: REPORT_READY=yes, BASELINE_SYNC_READY=yes, GIT_READY=yes, PUSH_READY=yes, WORKTREES_CLEANED=yes. Sin ellas el hook reescribe a blocked.

project-architect opus · maxTurns 80

bootstrap

task-planner sonnet · maxTurns 60

bootstrapmemoria persistente

Diseña el schema del registry. Memoria persiste entre proyectos: aprende qué patrones de Coverage Registry funcionan mejor.

main-orchestrator opus · maxTurns 150

sesión principal

Es la sesión de Claude Code, no un subagente invocable. Sólo este agente puede invocar a otros con Agent.

Anti-patrón: ejecutar claude plano y luego invocar main-orchestrator como subagente. Un subagente no puede orquestar otros. .claude/settings.json → "agent": "main-orchestrator" lo arranca correctamente.

Por qué NO lleva tools: en frontmatter

Omitir tools: hace que herede todas las herramientas disponibles, incluidos MCPs y la propia Agent. Si pones una lista, se convierte en allowlist y un MCP nuevo no llegaría.

deployer sonnet · maxTurns 80

worktree-sesiónsólo Phase 5

Sólo en release. Corre en el mismo checkout que el resto de la sesión worker (no abre un worktree propio). Exige closer committed previo, dry-run, plan de rollback. Trailer: deployed | planned | blocked | failed → done | blocked.

07Memoria por agente · Reflexion-style

Cada agente puede mantener un orchestrator-state/agent-memory/<agente>/MEMORY.md que sobrevive a /clear, reinicios y slices. Inspirado en Reflexion: el agente apunta lo que aprende, los bugs recurrentes, las decisiones que funcionaron.

Qué guarda cada uno

  • planner — patrones de task pack útiles
  • developer — convenciones del proyecto, gotchas del stack
  • validator — defectos arquitectónicos recurrentes
  • debugger — bugs vistos y causas raíz
  • official-docs-researcher — cache de versiones verificadas
  • task-planner — patrones de Coverage Registry (persistente entre proyectos)

Compactación segura

Cuando una memoria crece: scripts/compact-agent-memory.py --agent developer --apply. Archiva el original a archive/MEMORY.full.<ts>.md antes de reescribir un índice corto. Nunca pierde info histórica.

08Worktrees aislados (a nivel sesión, no subagente)

Versión actual: ningún subagente lleva isolation: worktree. El aislamiento Git por TASK_ID ocurre una vez por terminal worker, antes de que Claude Code arranque. Así todos los subagentes de la slice (developer, validator, tester, debugger, closer) inspeccionan y editan el mismo checkout.

Quién crea el worktree

El nuevo script ./scripts/ensure-task-worktree.sh. Lo invoca /next-wave dentro del bloque copy/paste que imprime, antes de lanzar claude --agent main-orchestrator.

  1. Lee git_workflow de STACK_PROFILE.yaml.
  2. Si git_workflow: push-to-main (o alias direct-main) → imprime la raíz canónica del repo y exige rama main. No abre worktree extra.
  3. Si git_workflow: pr-flow → crea (o reusa) la rama dev/<TASK_ID> en un worktree en <root>-worktrees/<TASK_ID> (configurable con CLAUDE_TASK_WORKTREES_DIR). Imprime esa ruta para que el terminal haga cd antes de arrancar Claude.

Modos de invocación

./scripts/ensure-task-worktree.sh P02-S03-T001          # crea/reusa y imprime la ruta
./scripts/ensure-task-worktree.sh --check-current P02-S03-T001
                                                          # gate: TASK_WORKTREE_READY: yes|no
./scripts/ensure-task-worktree.sh --print-root            # raíz canónica (no la del worktree actual)

Por qué session-level y no subagente

  • Validator/tester ven lo que escribió el developer — antes, si cada subagente abría su propio worktree, el validator inspeccionaba un árbol distinto al del developer.
  • Un solo git por slice — el closer commitea y hace push desde el mismo checkout donde se trabajó. Sin saltar entre worktrees.
  • Paralelismo entre terminales sigue intacto — cada terminal worker tiene SU propio TASK_ID y SU propio worktree. Dos terminales en paralelo no se pisan.

Por qué importa

  • Paralelismo seguro — dos slices distintas nunca tocan el mismo árbol físico.
  • Rollback gratis — borrar el worktree (cleanup-worktrees.sh) descarta los cambios sin tocar el resto.
  • Main siempre limpio en push-to-main; rama por task en pr-flow — el flujo Git encaja con el patrón declarado en STACK_PROFILE.
./scripts/cleanup-worktrees.sh --apply --task <TASK_ID> nunca borra worktrees dirty, ni el primario, ni ramas main/master. El closer lo invoca tras el push.

09La carrera silenciosa: validator ‖ tester

Dos agentes terminan a la vez y quieren mutar el registry. Sin tratamiento, sería race condition.

main-orchestrator │ ├─ Agent: validator ────┐ └─ Agent: tester ────┤ paralelo │ ┌──────────────────────────────┘ ▼ SubagentStop hook ─► file_lock(registry.json) │ ├─ validator (info-only): │ escribe task.validator_outcome (metadata) │ NO toca task.status │ └─ tester (lifecycle): escribe task.status = ready_for_close / needs_debug; slice-verifier añade verified_pending_close antes del closer Schema dice quién es info-only. Hook lo lee. Resultado deterministico: tester gana lifecycle, validator gana metadata.

Esto está declarado en .claude/orchestrator-contract.json → trailer_schema.roles.validator.info_only = true. El hook lo lee y aplica la regla automáticamente.
