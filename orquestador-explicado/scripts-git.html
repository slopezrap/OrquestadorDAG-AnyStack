Scripts, bin y el cierre Git

La mecánica del orquestador vive en Python (.claude/bin/) y en shell scripts (scripts/). Aquí está cada uno clasificado, más el cierre Git completo.

01Dos carpetas, dos propósitos

.claude/bin/

Python puro. Implementa hooks, bootstrap, claim, integrity checks, baseline sync, contracts. Una librería compartida (common.py) provee file_lock, IO atómico, helpers del registry.

scripts/

Shell wrappers que leen STACK_PROFILE.yaml y delegan al Python. Unos pocos tienen lógica propia: git-workflow.sh, cleanup-slice-runtime.sh, slice-clean.sh, cleanup-worktrees.sh, enforcers de tokens.

Lógica propia importante

git-workflow.sh

cierre

Dispatcher productivo del cierre. Lee git_workflow de STACK_PROFILE. Antes de delegar:

  1. Rechaza plugins con git stash.
  2. Amenda automáticamente sólo trazas tardías (ledger.jsonl, bash-ledger.jsonl, runtime-state.json) con git commit --amend --no-edit. Cualquier otro path dirty aborta.

slice-clean.sh

housekeeping

Post-closer. Dry-run por defecto. Rota ledgers > 200KB a .gz, borra cachés regenerables, archiva handoffs antiguos. No toca código ni source-of-truth ni registry.

cleanup-worktrees.sh

housekeeping

Borrado seguro de worktrees post push. Dry-run por defecto. Nunca toca worktree primario, ramas main/master, ni worktrees dirty.

[Nuevo] Tras borrar el último worktree del TASK_ID, comprueba si el directorio padre <repo>-worktrees/ quedó vacío y lo elimina (línea "removed empty container: ..." en la salida). Si todavía hay otros worktrees vivos dentro, lo respeta.

ensure-task-worktree.sh

nuevopre-flight Git

Crea o localiza el worktree por-TASK_ID antes de arrancar Claude Code. Lee git_workflow de STACK_PROFILE.yaml:

  • push-to-main (alias direct-main) → no abre worktree extra; exige rama main e imprime la raíz canónica del repo.
  • pr-flow → crea/reusa la rama dev/<TASK_ID> en <root>-worktrees/<TASK_ID> (configurable con CLAUDE_TASK_WORKTREES_DIR) e imprime esa ruta.

Modos: invocación normal (imprime ruta), --check-current <TASK_ID> (gate: TASK_WORKTREE_READY: yes|no), --print-root (raíz canónica, no la del worktree actual). Lo invocan /next-wave en el bloque copy/paste, /next-slice y /verify-slice al arrancar, y el closer antes del commit.

check-design-tokens.sh

enforcer

Dispatcher stack-agnóstico. Invoca .claude/enforcers/<enforcer>.sh según STACK_PROFILE.

check_design_tokens.py / check_web_design_tokens.py

Linters context-aware de tokens visuales para Flutter y Web. Strippean comentarios y strings antes de buscar literales.

dev-restart.sh + profile.sh / run-all-tests.sh / setup-from-scratch.sh

Dispatchers stack-agnósticos que leen comandos del STACK_PROFILE. Contrato --soft | --check | --reset.

check-progress-updated.sh

Gate del validator: PROGRESS.md tocado en la slice activa.

compact-agent-memory.py

Compactación lossless de MEMORY.md por agente. Archiva original primero. Dry-run por defecto.

04El cierre Git en detalle

El paso más delicado: cerrar y commitear sin dejar el árbol sucio.

① Pre-flight Git con ensure-task-worktree.sh

Antes del commit, el closer ejecuta ./scripts/ensure-task-worktree.sh --check-current <TASK_ID>. En pr-flow debe estar en la rama/worktree dev/<TASK_ID>; en push-to-main debe estar en main. Si el checkout es incorrecto, el closer bloquea (no cambia de rama por su cuenta).

② Closer prepara los artefactos antes del commit

No hace git stash. Antes del commit, el árbol contiene:

  • Cambios del developer (código del slice)
  • Handoffs y evidence apendados
  • Report nuevo en reports/<TASK_ID>.md
  • Baseline actualizado en docs/product-baseline/
  • PROGRESS.md actualizado

③ Commit atómico en el checkout actual

git add -A + git commit -m "<mensaje>". El commit cae en la rama actual: main si el workflow es push-to-main, dev/<TASK_ID> si es pr-flow. Reglas:

  • Mensaje conciso con TASK_ID
  • SIN Co-authored-by: Claude
  • SIN git stash

③ ./scripts/git-workflow.sh hace el push

Lee STACK_PROFILE.yaml → git_workflow. Antes de delegar:

  • Verifica que el plugin destino no use git stash
  • Si quedaron trazas tardías (3 ficheros runtime), amenda con git commit --amend --no-edit
  • Si hay otros paths sucios, aborta
Cambio reciente · pr-flow es ahora el default. Los templates en docs/templates/*/STACK_PROFILE.template.yaml traen git_workflow: pr-flow literal (no {{git_workflow}} placeholder). El prompt para ChatGPT también lo indica como default. push-to-main ahora es la excepción, sólo para proyectos single-developer single-slice sin PRs.

④ Plugin push-to-main.sh (default)

#!/usr/bin/env bash
set -euo pipefail
current_branch=$(git symbolic-ref --short HEAD)
if [ "$current_branch" != "main" ]; then
  echo "GIT_WORKFLOW_READY: no" >&2
  exit 1
fi
git push origin main
echo "GIT_WORKFLOW_READY: yes"
echo "PUSH_READY: yes"

El closer parsea esta salida para emitir GIT_READY: yes y PUSH_READY: yes.

⑤ Plugin pr-flow.sh (default · rebase + auto-merge)

Ahora hace fetch + rebase + push + create PR + PR merge controlado tras verify válido y /closer manual. Estrategia:

  1. [Nuevo] Pre-push rebase: git fetch <remote> main + git rebase <remote>/main. Si main avanzó mientras la slice estaba viva (cosa común con paralelismo), integra esos commits antes del push. Evita "PR mergeable: CONFLICTING" en la mayoría de los casos.
  2. Si el rebase tiene conflicto real → aborta limpiamente con git rebase --abort, emite REBASE_CONFLICT: yes + GIT_WORKFLOW_READY: blocked + comandos para resolver manualmente. Closer sale con exit 4, NO empuja nada a medias.
  3. git push -u --force-with-lease origin <branch> (force-with-lease porque el rebase reescribe SHAs; seguro contra pushes concurrentes)
  4. gh pr create --fill (o reusa el PR del branch si ya existe)
  5. gh pr merge --squash --delete-branch --admin → merge inmediato bypass de checks (porque /verify-slice ya verificó humanamente)
  6. Si --admin falla (no eres admin) → fallback a --auto (cola hasta que pasen required checks de GitHub)
  7. Si los dos fallan → deja el PR creado y emite MERGED: no con URL para revisión manual

El workflow Git del cierre se vuelve automático una vez que tú ejecutas /closer <TASK_ID> tras un /verify-slice verified: tu rama queda commiteada y mergeada a main sin que abras ningún PR a mano.

Salida machine-readable: GIT_WORKFLOW_READY, PUSH_READY, PR_READY, MERGED: yes | auto-queued | no, PR_URL: <url>.

Requisitos en tu Mac: gh auth status autenticado (`gh auth login` la primera vez), --admin requiere ser admin del repo, --auto requiere que el repo tenga auto-merge habilitado.

05Por qué dos ledgers separados

Problema sin separar: git push es Bash → dispara PostToolUse → hook_update_ledger.py escribe entrada. Si fuera versionado, el árbol acabaría dirty justo después del push, rompiendo el invariante "tras push, working tree limpio".

Solución

  • ledger.jsonl → Write/Edit/MultiEdit/NotebookEdit. Versionado por Git. Captura cambios reales.
  • bash-ledger.jsonl → Bash. Ignorado por Git. Las invocaciones shell post-commit no re-ensucian.

Y si algún PostToolUse de Bash toca runtime-state.json, git-workflow.sh hace amend --no-edit antes del push para integrar esa traza tardía.
