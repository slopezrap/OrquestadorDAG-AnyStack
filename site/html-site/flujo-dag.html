Reglas del scheduler

Join se desbloquea cuando todos sus predecessors están done

No con uno solo. Si A cierra y B no, el join sigue blocked. claim_task.py deniega el claim aunque el usuario lo intente manualmente.

Conflict group y Write set serializan

Dos slices independientes en el grafo (no comparten depends_on) pueden serializarse igualmente si pisan el mismo router, migración o fichero global declarado en STACK_PROFILE.

claim_task.py repite validaciones bajo lock

Aunque alguien intente saltar /next-wave con export CLAUDE_ACTIVE_TASK_ID=… manualmente, el claim falla si las deps no están done o hay conflicto activo.

/phase-gate bloquea avance entre fases

Si quedan tasks no done, journeys sin verificar, evidencia ausente o follow-ups bloqueantes, la fase no avanza aunque haya nodos ready en la siguiente.

Lock order — registry primero, runtime-state después

flowchart TD
    L1[1 lock registry.json]
    L2[2 modificar registry]
    L3[3 lock runtime-state.json]
    L4[4 modificar runtime-state]
    L5[5 release runtime-state]
    L6[6 release registry]
    L1 --> L2 --> L3 --> L4 --> L5 --> L6
Wave topológica. El bootstrap calcula topological_levels agrupando nodos por nivel de profundidad. Cada nivel es una "wave" — todos los nodos de la misma wave se pueden ejecutar en paralelo si Conflict group y Write set lo permiten.
