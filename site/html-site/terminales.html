Coordinación entre terminales

Qué pasa cuando una task termina.

No hay notificación push viva. La coordinación se hace por estado en disco, locks POSIX y /next-wave. Cada terminal exporta su CLAUDE_ACTIVE_TASK_ID y los hooks escriben con scope correcto.

Secuencia de eventos

EVENTO 01
Terminal A cierra una task. El closer emite trailer; el hook capture_subagent_stop valida enums, report, git, push y cleanup.
EVENTO 02
Hooks escriben bajo lock. registry.json, runtime-state.json y ledger.jsonl se actualizan con el TASK_ID correcto. Orden registry → runtime-state.
EVENTO 03
Se promueven successors. promote_ready_tasks recorre el DAG: si todas las deps están done, el nodo pasa a ready.
EVENTO 04
Terminal B no se interrumpe. Cuando quiera más trabajo, ejecuta ./scripts/next-wave.sh y obtiene la nueva frontera.
EVENTO 05
Nuevo claim seguro. Exporta CLAUDE_ACTIVE_TASK_ID y CLAUDE_TASK_PACK, lanza /next-slice. Claim atómico bajo lock.

Comandos por terminal

1 · Liberar el contexto del slice anterior

unset CLAUDE_ACTIVE_TASK_ID CLAUDE_TASK_PACK

2 · Pedir la siguiente wave segura

./scripts/next-wave.sh --limit 4

3 · Exportar el contexto del nuevo slice

export CLAUDE_ACTIVE_TASK_ID=Pxx-Syy-Tzzz \
       CLAUDE_TASK_PACK=orchestrator-state/tasks/task-packs/Pxx-Syy-Tzzz.md

4 · Arrancar el slice dentro de Claude Code

/next-slice Pxx-Syy-Tzzz

Lo que evita el motor

Doble claim del mismo nodo

El lock POSIX en registry.json serializa los claims. El segundo terminal que intente claim_task.py sobre la misma task obtiene denied.

Escrituras cruzadas entre TASK_IDs

El hook write_scope_guard deniega Write/Edit/MultiEdit que toquen handoffs, evidence o reports de un TASK_ID distinto al activo.

Mutaciones a estado derivado

Editar a mano registry.json, task-dag.json, runtime-state.json o ledger.jsonl bloqueado por hook. Solo bootstrap/claim/hooks pueden tocarlos.

Cierre falso de slice

El closer no puede emitir NEXT_STATUS: done si falta REPORT_READY, BASELINE_SYNC_READY, GIT_READY, PUSH_READY o WORKTREES_CLEANED en yes.

Saltarse el journey gate

Si runtime-state.pending_journey_verifications no está vacío, DAG-only difiere sólo las tasks que referencian esos journeys. Las ramas independientes pueden seguir; las tasks afectadas deben verificar el journey inline en §5.bis o aparte con /verify-journey.

Conflict entre stack-specific paths

Los Conflict group y Write set se declaran en función del STACK_PROFILE.yaml. Si dos slices tocan el mismo router/migración del stack declarado, se serializan automáticamente aunque sean independientes en el DAG.
