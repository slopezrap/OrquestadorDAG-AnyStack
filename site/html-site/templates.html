Generación de docs

Tres perfiles de template, mismo runtime.

Los templates producen los cinco documentos de docs/source-of-truth/: instrucciones.md, *_TECHNICAL_GUIDE.md, *_IMPLEMENTATION_CHECKLIST.md, STACK_PROFILE.yaml y UX_CONTRACT.md. El motor DAG es idéntico en los tres casos — lo único que cambia es el esfuerzo para generar la primera versión y si arrastra baseline acumulativo o no.

Apps grandes con baseline

large-with-base

docs/templates/large-with-base/*.template.{md,yaml}

  • existing baseline / v1 / v2 / vN acumulativo
  • Library Discovery Pass amplio
  • Phases por lanes o pantallas
  • Conflict group + Write set obligatorios
  • Verification Data Contract real/provided
  • ~80-150 slices realistas
Cuándo usar. Productos con baseline ya construido, o apps complejas con auth + datos + AI + admin + GDPR. La existing baseline incluida tiene 84 slices y 8 journeys reales.
Apps grandes desde cero

large-without-base

docs/templates/*.template.{md,yaml} · sin existing baseline

  • Templates large sin heredar existing baseline
  • Sin tablas/endpoints/journeys históricos
  • Phases pequeñas, DAG explícito
  • Stack Profile + UX Contract obligatorios
  • Coverage Registry empieza limpia
  • ~30-80 slices del primer incremento
Cuándo usar. Producto grande desde cero sin un baseline previo del que partir. Punto medio entre large-with-base (baseline real) y minimal (scope pequeño).
Apps pequeñas

minimal

docs/templates/minimal/*.template.{md,yaml}

  • 2-4 phases
  • 3-8 tasks totales
  • 1-2 journeys reales multi-pantalla
  • Mismo DAG explícito
  • Sin existing baseline ni Library Discovery
  • On-ramp drásticamente reducido
Cuándo usar. Apps de 5-15 pantallas sin baseline. MVP, herramientas internas, prototipos productivos. MiniNotes (5 tasks, 1 journey) está validado en CI.

Los 5 documentos source-of-truth

Cada perfil produce el mismo set canónico. AnyStack añade STACK_PROFILE y UX_CONTRACT al contrato contrato histórico de cinco docs.

Documento Owner Qué declara
instrucciones.mdProductoScope, business rules, Journey Coverage Matrix
UX_CONTRACT.mdProductoPersonas, pantallas, estados UI requeridos, verificación visual
*_TECHNICAL_GUIDE.mdTech leadArquitectura, contratos, endpoints, DB, ADR, Verification Data Contract
STACK_PROFILE.yamlTech leadFrameworks, paths, comandos, health, design-tokens enforcer, git workflow
*_IMPLEMENTATION_CHECKLIST.mdPM / Tech leadPhases, Coverage Registry con DAG (Depends on, Conflict group, Write set)

Proceso común

PASO 01
Pasa a ChatGPT el prompt maestro. docs/prompts/PROMPT_SOURCE_OF_TRUTH_DAG.md + el contexto real del producto.
PASO 02
Elige perfil. large-with-base si heredas existing baseline. large-without-base si es producto grande desde cero. minimal si es app pequeña sin existing baseline.
PASO 03
Rellena los 5 docs completos. instrucciones.md, *_TECHNICAL_GUIDE.md, *_IMPLEMENTATION_CHECKLIST.md, STACK_PROFILE.yaml y UX_CONTRACT.md en docs/source-of-truth/.
PASO 04
Bootstrap + checkers. Genera registry, work-items, DAG y vistas derivadas. Falla rápido si los docs están incompletos o si STACK_PROFILE referencia un enforcer/git-workflow inexistente.
PASO 05
/next-wave y construye slices. A partir de aquí, el flujo es idéntico para los tres perfiles.

Comandos de bootstrap

python3 -B -S .claude/bin/bootstrap_source_of_truth.py --validate-only
python3 -B -S .claude/bin/bootstrap_source_of_truth.py --refresh
./scripts/check-task-dag.sh --strict
./scripts/check-journey-matrix.sh --strict
./scripts/check-wiring-contract.sh --strict --require-new-template-columns
Los tres perfiles comparten runtime. Hooks, locks, agentes, gates, journey verification y closer son idénticos. Solo cambian dos cosas: la cantidad de scope inicial y si arrastras baseline acumulativo.
